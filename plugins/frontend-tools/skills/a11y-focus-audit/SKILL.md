# A11Y Focus Audit

## 무엇을 / 언제 (한 줄)
프론트엔드 코드나 실행 화면의 접근성 문제와 Tab 포커스 이동 문제를 리뷰할 때 사용한다.

## 제약 (MUST DO / MUST NOT DO) — 스텝 위에 둔다
- ✅ 작업 전 대상 파일, 라우트, 실행 방법을 repo에서 먼저 찾는다.
- ✅ 코드 점검과 실제 키보드 포커스 이동 확인을 분리해서 보고한다.
- ✅ 파일/셀렉터/심각도/수정안을 포함한 이슈 목록으로 출력한다.
- ⛔ 실제 브라우저 확인 없이 Tab 이동이 정상이라고 단정하지 않는다.
- ⛔ 접근성 검토 범위를 벗어난 디자인 취향 문제를 섞지 않는다.
- ⛔ 자동 수정이나 대규모 리팩터링을 검토 요청에 포함하지 않는다.
- ⛔ 출력은 cwd 상대 경로 + `mkdir -p` 가드로만 한다. 채팅 덤프 금지.

## 핵심 워크플로우
1. 대상 파악: 사용자가 준 파일, 화면, 라우트, viewport, 실행 명령을 확인하고 부족하면 repo에서 찾는다. — 입력: 요청/파일/라우트. 출력: 점검 대상과 실행 가능 여부.
2. 코드 점검: 시맨틱 요소, 폼 label, alt, aria, 숨김 텍스트, focus style, keyboard handler, disabled 상태를 본다. — 입력: 관련 HTML/JS/TS/CSS. 출력: 코드 기반 접근성 의심 지점.
3. 브라우저 점검: 실행 가능한 화면이면 Playwright 또는 browser 도구로 Tab/Shift+Tab 순서, visible focus, modal/dropdown trap, skip link를 확인한다. — 입력: 실행 중인 URL. 출력: 실제 포커스 이동 증거와 실패 지점.
4. 판정: 코드 문제와 브라우저 재현 문제를 합쳐 severity를 매긴다. — 입력: 코드 점검 결과와 브라우저 결과. 출력: 중복 제거된 이슈 목록.
5. 검증: 각 finding에 파일/셀렉터, 문제, 영향, 최소 수정안, 검증 방법이 있고 브라우저 미확인 항목은 미확인으로 표시했는지 를 확인한다.
6. 출력: 산출물을 cwd 상대 경로에 둔다.

## 출력 위치 (필수)
- ✅ Severity별 이슈 목록: Critical/High/Medium/Low
- ✅ 각 이슈의 위치: 파일 경로 또는 화면 셀렉터
- ✅ 각 이슈의 최소 수정안과 재검증 방법
- ✅ 검증 실행 여부: 코드 점검, 브라우저 Tab 점검, 미실행 사유
