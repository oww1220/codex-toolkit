# /design-brief — 제품·사용 환경 인터뷰

> AI Design Director 9단계 파이프라인의 **1단계**. 디자인 결정을 시작하기 전에,
> *제품이 무엇이고 / 누가 / 어떤 환경에서 / 무엇을 자주 하며 / 어떤 인상을 원하는지* 를
> 사용자와의 인터뷰로 확정한다. 산출물은 다음 단계(`/design-translate`)의 단일 입력이다.

## 정신 (BUILD CONTRACT §0 6원칙)

- **생성보다 판단**: 색/폰트/레이아웃을 *아직 정하지 않는다*. 이 단계는 "사실 수집"이지 "디자인 결정"이 아니다. 토큰·컴포넌트·스타일 제안 금지.
- **말보다 선택을 신뢰**: 추상 표현("미니멀하게")은 그대로 받아 적되, 모호하면 선택형 보기를 제시해 *판단*하게 한다.
- **모든 결정엔 이유**: 사용자 표현 원문은 보존하고, AI 해석은 별도 칸에 분리한다. 확정(confirmed) vs 추정(assumed)을 항상 구분한다.

---

## 1. 목적

| 항목 | 내용 |
|------|------|
| **입력** | 자연어 설명 / 기존 PRD·기획서 / 소스코드 / 홈페이지 URL / 로고·브랜드 자산 / 참고 이미지 / 기존 디자인 시스템 (있는 만큼) |
| **출력** | `design/DESIGN-BRIEF.md` (한국어, `templates/DESIGN-BRIEF.template.md` 골격을 채움) |
| **다음 단계** | `/design-translate` (이 브리프 → 디자인 언어 번역) |

이 단계의 성공 기준: 디자이너가 처음 보더라도 *무엇을 만드는지·누구를 위한 건지·무엇이 중요한지*를
브리프만 읽고 이해할 수 있어야 한다. 디자인 *결정*은 이후 단계의 몫이다.

---

## 2. 단계별 플레이북

### Step 0 — 사전 수집 (질문하기 *전에* 먼저 읽는다)

> **원칙: 이미 알 수 있는 것은 다시 묻지 않는다.** 인터뷰 피로를 줄이는 핵심.
> 사용자가 던진 입력에서 추출 가능한 사실을 먼저 수집하고, *빈칸만* 질문한다.

수집 순서와 도구:

1. **프로젝트 안에 이미 단서가 있는가** — `Glob` 으로 탐색:
   - PRD/기획: `**/{PRD,prd,README,readme,기획}*.md`, `docs/**/*.md`
   - 소스: `package.json`, `**/routes/**`, `**/pages/**`, `**/app/**`, `**/components/**`
   - 기존 디자인: `**/{DESIGN,tokens,theme,tailwind.config}*`, `design/**`
2. **단서가 있으면 `Read`** 로 핵심만 읽어 제품 정의·주요 화면·기능을 파악한다.
3. **키워드 확인이 필요하면 `Grep`** (예: 도메인 용어, 화면명, 역할명).
4. **홈페이지/참고 URL 을 사용자가 줬으면 `WebFetch`** 로 제품 설명·톤·기존 인상을 수집한다.
5. **로고·참고 이미지를 줬으면 `Read`** 로 로드해 색·분위기를 *관찰만* 한다(아직 토큰화하지 않음).

> 산출물 메모: 수집한 사실은 모두 *근거(source)* 로 기록한다 — "PRD 3장에서 확인", "홈페이지 hero 문구",
> "사용자 발언" 등. 브리프의 모든 항목은 `확정`(근거 있음)과 `추정`(AI 가 메움)으로 나뉜다.

수집 후, **무엇을 이미 알았고 무엇이 비었는지** 1줄 요약을 사용자에게 보여준 뒤 질문을 시작한다.

---

### Step 1 — 인터뷰 (AskUserQuestion 으로 *묶어서*)

> **한 번에 질문 과다 금지.** 5개 질문 영역을 한 번에 쏟지 말고,
> `AskUserQuestion` 한 호출당 관련 질문 2~4개씩 묶어 **클릭으로 답하게** 한다.
> 일반 텍스트로 "어떤 색을 원하세요?" 던지고 기다리는 방식은 금지.
> Step 0 에서 *이미 확정된 항목은 질문에서 제외*한다.

5개 질문 영역(모두 커버하되, 이미 안 것은 건너뛴다):

#### (A) 제품 정의
- 무엇을 만드는가 / 어떤 문제를 해결하는가 / 핵심 기능 1~3개 / 사용자가 얻는 핵심 가치.
- PRD·소스로 대부분 채워졌다면, **확인만** 받는다("제품을 X로 이해했는데 맞나요?").

#### (B) 사용자 정의
- 주요 사용자(역할/직군) / 디지털 숙련도 / 사용 환경(데스크톱 업무용·모바일·현장·매장 등) / 사용 빈도·1회 사용 시간.
- 보기 예: 숙련도 `처음/가끔/매일/전문가`, 환경 `사무실 데스크톱/이동 중 모바일/매장 태블릿/혼합`.

#### (C) 핵심 업무
- 가장 자주 하는 작업 / 가장 중요한 행동(이게 쉬워야 성공) / 절대 실수하면 안 되는 작업 / 많은 정보를 동시에 봐야 하는 화면 / 오래 집중하는 작업이 있는지.
- 이 영역이 **정보 밀도·테이블 vs 카드·내비게이션 구조**를 좌우하므로 가장 구체적으로 받는다.

#### (D) 브랜드 인상 (우선순위 선택)
- 다음 중 **우선순위**를 고르게 한다(다중 선택 + 순서): `신뢰 / 전문성 / 속도 / 친근함 / 차분함 / 실험성 / 고급스러움 / 인간적 / 기술적`.
- `AskUserQuestion` 으로 상위 2~3개를 고르게 하고, 1순위가 무엇인지 확인한다. (가짜 형용사 나열 대신 *선택*하게.)

#### (E) 선호·거부
- 좋아하는/싫어하는 사이트·앱(있으면 URL — Step 0 에서 `WebFetch` 가능) / 쓰고 싶은·피하고 싶은 색 / 보유 브랜드 자산(로고·폰트·컬러) / **반드시 피해야 할 디자인**.
- "반드시 피할 것"은 안티패턴 회피의 직접 근거가 되므로 빠뜨리지 않는다.

#### 모호할 때의 규칙
사용자가 "그냥 깔끔하게요" 처럼 모호하게 답하면, **선택형 예시**를 제시한다:
> "깔끔함에도 결이 있어요. 어느 쪽에 가깝나요?" → `여백 많고 절제된(에디토리얼) / 정보 빽빽하지만 정돈된(생산성) / 부드럽고 친근한 / 차갑고 기술적인`
이때도 `AskUserQuestion` 의 보기로 제시한다. 디자인 *제안*이 아니라 *방향 좁히기*임에 유의.

### Step 2 — 정리 및 작성

수집·인터뷰 결과를 `templates/DESIGN-BRIEF.template.md` 골격에 맞춰 `design/DESIGN-BRIEF.md` 로 작성한다.

작성 규칙:
- **사용자 표현 원문 보존** — 따옴표로 인용("오래 봐도 안 질리게"). 윤색·요약하지 않는다.
- **AI 해석 분리** — 원문 아래 별도 `해석:` 줄에 둔다. 둘을 섞지 않는다.
- **확정/추정 구분** — 각 항목 끝에 `[확정·근거: …]` 또는 `[추정]` 표기. 근거 없는 단정 금지.
- **디자인 결정 금지** — 색 hex·폰트명·radius 값·컴포넌트 스타일을 *여기서 정하지 않는다*. (그건 `/design-translate` 이후의 일.)
- **빈칸은 빈칸으로** — 못 받은 정보는 정직하게 `미정`으로 남긴다. 추측으로 채우지 않는다.

브리프 본문 구성(템플릿 골격):

```
# DESIGN-BRIEF — <프로젝트명>
생성: <date> · 단계: 1/9 design-brief · 다음: /design-translate

## 1. 제품 정의        (무엇 / 문제 / 핵심기능 / 핵심가치)
## 2. 사용자 정의       (주요사용자 / 숙련도 / 환경 / 빈도·시간)
## 3. 핵심 업무         (자주 / 가장 중요 / 실수 금지 / 정보 밀집 / 집중 작업)
## 4. 브랜드 인상       (우선순위 표 — trait + 순위)
## 5. 선호·거부         (좋아함 / 싫어함 / 색 선호·기피 / 보유 자산 / 반드시 피할 것)
## 6. 확정 vs 추정      (confirmed[] / assumed[] 요약)
## 7. 근거(sources)     (PRD/소스/URL/사용자 발언 출처 목록)
```

## 3. 산출물 경로 (BUILD CONTRACT §3 — 절대 고정)

| 파일 | 경로 |
|------|------|
| 디자인 브리프 | `design/DESIGN-BRIEF.md` |

> 사용자 프로젝트 루트에 `design/` 폴더가 없으면 만든다. 이 단계는 `design/` 폴더의 **첫 산출물**이다.

---

## 4. 품질 체크 (작성 직후 자가검증)

- [ ] 5개 질문 영역(제품/사용자/업무/인상/선호) 이 모두 다뤄졌다 (못 받은 것은 `미정` 명시).
- [ ] `AskUserQuestion` 으로 **묶어서** 물었고, 한 번에 질문을 과다 투척하지 않았다.
- [ ] Step 0 에서 PRD·소스·URL 로 알 수 있던 것을 **다시 묻지 않았다**.
- [ ] 사용자 표현 **원문**과 AI **해석**이 분리돼 있다.
- [ ] 모든 항목이 `[확정·근거]` 또는 `[추정]` 으로 표시돼 있다.
- [ ] 색 hex·폰트명·radius·컴포넌트 스타일 등 **디자인 결정을 하지 않았다**(이 단계는 사실 수집).
- [ ] 진부한 AI 기본값을 *제안하지 않았다* — 순수 `#000/#fff`, Tailwind 기본 blue/violet, 이모지 장식, 가짜 수치, 균등 카드 그리드 (BUILD CONTRACT §1 자가검열).
- [ ] 경로가 `design/DESIGN-BRIEF.md` 와 정확히 일치한다.

---

## 5. 다음 단계 안내

작성 완료 후 사용자에게 알린다:

> `design/DESIGN-BRIEF.md` 가 준비됐습니다. 다음은 **`/design-translate`** — 이 브리프의
> 표현("차분하게", "정보 많지만 안 복잡하게")을 구현 가능한 디자인 언어(visual tone, 정보 밀도,
> 타이포 방향, 금지 요소 등)로 번역하고 `design/design-profile.json` 을 만듭니다.

`design-translate` 가 `design/DESIGN-BRIEF.md` 를 입력으로 읽으므로, 다음 단계 진입 전
브리프의 **확정/추정 표기**가 채워졌는지 다시 확인한다. (추정이 너무 많으면 인터뷰를 한 번 더 권한다.)

## 참고 — 이전 단계 산출물 부재 처리

`/design-brief` 는 파이프라인의 **첫 단계**이므로 선행 산출물이 없다.
단, 사용자가 곧장 `/design-translate` 등 뒷 단계를 호출했는데 `design/DESIGN-BRIEF.md` 가 없으면,
그 단계가 이 커맨드를 먼저 돌리도록 유도한다(`/design-full` 오케스트레이션은 순서를 보장한다).
