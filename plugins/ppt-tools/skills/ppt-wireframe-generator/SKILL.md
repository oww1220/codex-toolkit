# PPT Wireframe Generator

## 무엇을 / 언제 (한 줄)
사용자가 PPT 와이어프레임 또는 화면 흐름용 슬라이드 초안을 요청할 때, 요구사항을 낮은 충실도 화면 구조로 정리한 뒤 Presentations 스킬로 .pptx를 만든다.

## 제약 (MUST DO / MUST NOT DO) — 스텝 위에 둔다
- ✅ 화면/페이지 목록, 사용자 흐름, 주요 UI 영역, CTA/상태/메모 수준을 먼저 정리한다
- ✅ 플러그인 통합 실행처럼 `ppt-planning-harness`와 함께 쓰는 요청은 `ppt-planning-harness`로 요구사항/검토 결과를 먼저 정리한 뒤 이 스킬로 PPTX를 생성한다
- ✅ 실제 deck 생성 전 Presentations 스킬을 로드하고 그 workspace, artifact-tool, QA 규칙을 따른다
- ✅ 와이어프레임은 저충실도 구조와 흐름 전달에 집중하되, 의미 구분용 색상은 기본 적용한다
- ✅ 색상은 사용자/어드민/공통 영역, CTA, 상태, 우선순위, 확인 필요 항목처럼 정보 구분에 쓰고 과한 장식은 피한다
- ✅ 화면 구조만으로 의도가 불명확하면 오른쪽 설명 영역에 목적, 주요 액션, 상태, 확인 필요 같은 상세 설명을 짧게 붙인다
- ✅ 산출물은 와이어프레임 겸 개발 지시 문서로 사용되므로 화면별 구현 의도, 데이터/상태/권한, 예외 케이스, 확인 필요 항목을 최대한 정확하고 추적 가능하게 남긴다
- ✅ 각 화면 슬라이드에는 필요한 경우 `Sequence`와 `Business Rules`를 짧게 포함해 화면 전환, API/데이터 호출 순서, 분기 조건, 금지/허용 규칙이 구현자가 바로 보이게 한다
- ✅ `테이블명세서`, `DB 명세서`, `테이블 정의서`, `컬럼 명세`, `스키마 명세서` 요청은 화면 와이어프레임과 별도 `Table Spec` 분기로 처리한다
- ✅ PPT Planning Harness 검토 결과는 별도 체크리스트 슬라이드가 아니라 각 슬라이드의 설명 레일, 상태/예외, 구현 메모, 확인 필요 영역에 나눠 반영한다
- ✅ 내부 검토 항목은 `State`, `Validation`, `API/Data`, `Access`, `Verify`, `Open Questions`처럼 짧은 실무형 라벨로 녹이고, 검토 항목명을 그대로 노출하지 않는다
- ✅ PPTX 생성/수정 후 `scripts/check_wireframe_deck.py <file.pptx>`와 PPT Planning Harness의 `check_planning_doc.py <file.pptx>` 검사를 실행하고, 실패하면 `FAIL:` 항목을 deck에 반영한 뒤 통과할 때까지 재검증한다
- ⛔ Presentations 스킬을 우회해 python-pptx, 임의 SVG 드로잉, 별도 렌더링 엔진으로 PPT를 만들지 않는다
- ⛔ Google Slides import나 네이티브 Slides 생성은 사용자가 별도로 요청하지 않는 한 포함하지 않는다
- ⛔ 브랜드 최종 디자인, 고충실도 mockup, 마케팅용 hero deck으로 범위를 넓히지 않는다
- ⛔ 출력은 cwd 상대 경로 + `mkdir -p` 가드로만 한다. 채팅 덤프 금지.

## 시각 품질 기준
- 16:9 슬라이드는 공통 마스터처럼 보이게 고정한다: 상단 헤더, 좌측 화면 캔버스, 우측 설명 레일, 하단 CTA/API/validation 요약
- 상단 헤더에는 화면명, route 또는 flow 위치, 상태 배지만 둔다. 긴 설명은 헤더에 넣지 않는다
- 좌측 화면 캔버스는 실제 UI에 가까운 입력창, 버튼, 탭, 리스트, 오류 문구, loading/disabled 상태를 배치하고 회색 박스 반복으로 때우지 않는다
- 우측 설명 레일은 슬라이드 폭의 25~30%를 쓰고 `목적`, `Sequence`, `Business Rules`, `개발 메모`, `상태/예외`, `확인 필요`, `Verify focus` 중 필요한 3~5개만 짧게 둔다
- 하단 요약은 CTA, API/Data, Validation을 한 줄 카드나 얇은 바 형태로 정리한다. 길면 슬라이드를 나눈다
- 색상은 배경 전체가 아니라 CTA, 상태, 확인 필요, 권한/역할 구분에만 작은 면적으로 쓴다
- 기본 팔레트는 밝은 배경 + 진한 본문 + 브랜드 강조색 1개 + 경고색 1개로 제한한다
- 제목, 섹션 라벨, 본문, 상태 태그의 폰트 계층을 명확히 나누고 긴 문장은 글자를 줄이지 말고 내용을 줄인다
- 상태 배지, flag 딱지, 조건 라벨은 고정 높이/패딩을 쓰고 텍스트를 가로·세로 중앙정렬한다. 특히 상하 중앙정렬은 중요한 가독성 기준이므로 PPT 텍스트 박스의 세로 정렬을 middle로 고정하고, 줄간격/내부 여백 때문에 치우치면 박스 높이나 패딩을 조정한다. 문구가 길면 글자를 줄이지 말고 짧은 명사로 바꾸거나 설명 레일로 보낸다
- 배지류는 헤더 상태, 화면 내부 상태, 흐름 조건처럼 역할별 위치를 고정하고 산발적으로 흩뿌리지 않는다
- 카드 안 카드, 과도한 pill/badge, 작은 박스와 라벨 남발, 배경 전체 색칠은 피한다
- 내부 검토용 표현인 `22개 검토 항목`, `22가지 요구사항`, `Planning Harness 체크리스트`, `Executive Summary`, `Final Document`는 슬라이드에 쓰지 않는다

## 가독성 기준
- 한 슬라이드는 한 화면 또는 한 흐름만 설명하고, 읽는 순서는 좌상단에서 우하단으로 자연스럽게 흐르게 만든다
- 화면 안 라벨은 핵심 명사 중심으로 짧게 쓰고, 설명 문장은 오른쪽 설명 레일로 분리한다
- 오른쪽 설명 레일은 목적, 주요 액션, 상태/예외, 확인 필요를 3~5개 bullet로만 정리한다
- `Sequence`는 3~5단계 텍스트 화살표나 작은 단계 블록으로 표현하고, `Business Rules`는 2~4개 짧은 규칙으로 제한한다. 길어지면 별도 흐름형 슬라이드로 나눈다
- 본문이 많아지면 슬라이드를 나누고, 작은 글씨로 한 장에 밀어 넣지 않는다
- 흐름형 슬라이드는 각 단계 이름, 전환 조건, 최종 CTA만 남기고 부가 설명은 별도 화면 설명형 슬라이드로 보낸다

## Table Spec 분기
- 테이블명세서 요청이면 좌측 화면 캔버스 레이아웃을 쓰지 않고, 테이블 요약 + 컬럼 명세 + 관계/제약 + 확인 필요 중심의 명세형 슬라이드로 만든다
- 기본 슬라이드는 `테이블 목록/요약`, `테이블별 컬럼 명세`, `PK/FK/관계`, `인덱스/제약조건`, `상태값/enum`, `관련 API/화면`, `확인 필요` 중 필요한 흐름으로 구성한다
- 테이블 정의서의 마지막 슬라이드는 `ERD`로 구성하고, 주요 테이블과 PK/FK 관계를 한 화면에 요약한다
- 컬럼 표에는 최소 `컬럼명`, `타입`, `필수`, `기본값`, `키`, `설명`을 둔다
- 한 슬라이드에 컬럼이 많으면 글자를 줄이지 말고 테이블별 또는 필드 그룹별로 나눈다
- ERD가 과밀하면 핵심 도메인 관계만 남기고 상세 컬럼은 각 테이블 명세 슬라이드로 보낸다
- 개인정보, 권한, 민감 데이터 컬럼은 작은 배지로 표시하되, 배지 텍스트는 가로·세로 중앙정렬한다
- 불확실한 타입, 관계, 기본값, 인덱스, 제약조건은 확정하지 말고 `확인 필요`로 표시한다

## 핵심 워크플로우
1. 입력을 와이어프레임 brief로 압축한다 — 사용자 요청 -> 화면/슬라이드 목록, 사용자 흐름, 각 화면의 핵심 영역과 상태
2. 정보가 부족하면 핵심 결정만 묻는다 — 누락된 대상 화면, 흐름, 산출 경로, 주석 수준 -> 최소 질문 또는 합리적 기본값
3. Presentations 스킬을 로드한다 — 확정 brief -> Presentations의 workspace, artifact-tool, 출력, QA 규칙
4. 저충실도 wireframe deck을 생성한다 — brief + Presentations 규칙 -> 화면 흐름 중심 .pptx
   - 요청이 테이블명세서 계열이면 `Table Spec 분기`를 사용하고, 화면 캔버스/CTA 중심 구성을 테이블 구조/제약/관계 중심 구성으로 바꾼다
   - 기본 레이아웃은 화면 단독형, 화면 설명형, 흐름형 중 하나를 선택하되, 가능하면 상단 헤더 + 좌측 화면 캔버스 + 우측 설명 레일 + 하단 요약 구조를 유지한다
   - 화면 단독형: 큰 화면 캔버스 + 하단 CTA/API/validation 요약으로 구성한다
   - 화면 설명형: 좌측 화면 캔버스 + 우측 목적/개발 메모/상태/예외/확인 필요 메모로 구성한다
   - 흐름형: 3~5개 화면 썸네일 + 전환 화살표 + 조건 라벨로 구성한다
   - 기본 색상 적용: 사용자=파랑 계열, 어드민=보라 계열, CTA=오렌지 계열, 확인 필요=노랑 계열처럼 의미가 드러나는 팔레트를 사용한다
   - 색상은 배경 전체가 아니라 라인, 라벨, CTA 박스, 상태 태그 등 작은 면적에 우선 적용한다
   - 상세 설명이 필요한 화면은 오른쪽 설명 레일을 두고 목적, 주요 액션, 상태/예외, 확인 필요를 3~5개 짧은 bullet로 정리한다
   - 화면별 구현 이해에 필요한 경우 `Sequence`에는 사용자 액션 -> 화면 상태 -> API/Data -> 결과/오류 흐름을, `Business Rules`에는 권한, 유효성, 상태 전이, 금지 조건을 짧게 적는다
   - 개발 지시 문서로 이어질 수 있게 API/데이터 출처, 권한 분기, 빈 상태/오류 상태, 구현 확인 포인트가 있으면 페이지 안에 짧게 주입한다
   - 불확실한 내용은 확정처럼 쓰지 말고 `확인 필요`로 표시한다
   - 오른쪽 설명은 화면 박스보다 보조 정보로 보이게 만들고, 텍스트가 많으면 내용을 줄인다
5. 검증: Presentations의 최종 preview/contact-sheet QA를 완료하고, 각 슬라이드에 화면명 또는 테이블명이 있으며, 화면 흐름 PPT는 흐름상 위치, 주요 UI 영역, CTA 또는 상태 메모와 의미 구분 색상이 있고, 테이블명세서 PPT는 컬럼 타입/필수/키/설명, 관계/제약, 민감정보/권한/상태값 표시가 있으며 마지막에 ERD 슬라이드가 있고, 개발 지시 문서로 필요한 구현 메모가 페이지별로 녹아 있고, 필요한 화면에는 `Sequence`와 `Business Rules`가 짧고 추적 가능하게 포함되어 있으며, 불확실한 내용은 `확인 필요`로 표시되어 있으며, 필요한 경우 오른쪽 상세 설명이 있고, 여백/정렬/폰트 계층/색상 일관성이 유지되며, 배지/flag/조건 라벨의 텍스트가 박스 안에서 가로·세로 중앙정렬되고 세로 정렬이 middle이며 역할별 위치가 일관되며, 읽는 순서가 명확하고, 한 슬라이드에 정보가 과밀하지 않으며, overlap/wrapping/clipping 문제가 없고, 내부 검토 항목명이 슬라이드에 그대로 노출되지 않았음을 확인한다. 이어서 `scripts/check_wireframe_deck.py <file.pptx>`와 PPT Planning Harness의 `check_planning_doc.py <file.pptx>` 검사를 실행하고, 실패하면 `FAIL:` 항목을 반영해 deck을 수정한 뒤 다시 실행한다.
6. 출력: 산출물을 cwd 상대 경로에 둔다.
