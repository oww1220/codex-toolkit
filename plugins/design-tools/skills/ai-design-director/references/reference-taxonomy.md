# 레퍼런스 분류 체계 (Reference Taxonomy)

> `/design-research` 가 레퍼런스를 탐색·분류·평가할 때 참조하는 지식 베이스. 산출물
> `design/research/REFERENCE-REPORT.md` 와 `design/research/references.json`(Reference Candidate[])의
> 근거가 된다. (BUILD CONTRACT §11 / §18)

---

## 0. 가장 중요한 원칙 — 두 축을 분리한다

**예쁜 마케팅 스타일이 업무용 SaaS에 맞는다는 보장은 없다.** 그래서 레퍼런스는 **하나의 서비스를
통째로 베끼지 않고**, 두 개의 독립된 축으로 분해해서 본다:

| 축 | 무엇을 가져오나 | references.json `category` |
|----|----------------|----------------------------|
| **Visual Style** | 색상·타이포·표면·여백·모서리·아이콘·이미지·분위기 | `visual-style` |
| **UX Pattern** | 내비게이션·정보구조·검색·필터·목록·상세·폼·상태·반복업무 흐름 | `ux-pattern` |
| (둘을 한 후보에 동등 결합) | 위 둘을 모두 강하게 가진 후보 | `hybrid` |

→ 시각 스타일은 A 서비스에서, UX 구조는 B 서비스에서 가져올 수 있다. 후보(candidate)는
이 두 축 중 무엇을 대표하는지 `category` 로 명시한다. 최종 합성은 `/design-select` 에서
요소 단위(15종)로 조합된다.

## 1. Visual Style Reference — 구성 요소 분해

레퍼런스에서 "시각 스타일"을 볼 때 아래 항목을 각각 관찰·기록한다.

| 요소 | 관찰 포인트 | references.json 으로 가는 곳 |
|------|------------|------------------------------|
| **색상** | 베이스/서피스/텍스트/보더 뉴트럴, accent 개수와 채도, semantic 색, 다크 대응 | keywords, apply, expectedImpression |
| **타이포** | display/body 폰트 family, weight 대비(800+ vs 400~500), 사이즈 스케일, 위계 강도 | apply / doNotApply |
| **표면 질감** | flat / border / elevation / blur, 그림자 단계 | apply |
| **선/면 비율** | 구분선 중심 vs 면(배경톤) 중심, 보더 두께 | apply |
| **여백** | 밀도(조밀↔여유), 리듬, 그룹 간격 | recommendedFor |
| **모서리** | radius 스케일, pill 사용처, 일관성 | apply |
| **아이콘** | 라인/솔리드, 굵기, 의미 사용 vs 장식, 통일 세트 여부 | apply / doNotApply |
| **이미지 스타일** | 실사/일러스트/추상, 처리(ken-burns·dim), 제품 데이터 노출 | apply |
| **분위기** | 차분/대담/따뜻/기술적 등 1~2단어 인상 | expectedImpression |

**기록 규칙**: 각 요소에서 *가져올 것(apply)* 과 *가져오지 않을 것(doNotApply)* 을 분리한다.
"이 사이트는 색은 좋은데 타이포는 우리 제품엔 과하다" 처럼.

## 2. UX Pattern Reference — 구성 요소 분해

"UX 구조"를 볼 때 아래 항목을 관찰한다. 시각이 예뻐도 UX 구조가 제품에 안 맞을 수 있다.

| 요소 | 관찰 포인트 |
|------|------------|
| **내비게이션** | left-sidebar / top-bar / command-search / bottom-tab, 계층 깊이, 전환 방식 |
| **정보구조(IA)** | 무엇이 최상위인가, 그룹핑 논리, 드릴다운 경로 |
| **검색** | 전역 검색 위치, 즉시검색, 범위 지정 |
| **필터** | 필터 위치(상단/사이드), 다중 필터, 저장된 뷰 |
| **목록·테이블** | 행 밀도, 정렬, 컬럼 커스터마이즈, 일괄 선택·작업, 페이지네이션 vs 무한스크롤 |
| **상세(detail)** | 마스터-디테일, 패널 vs 전체페이지, 인라인 편집 |
| **폼** | 단일/다단계, 인라인 검증, 레이블 위치, 저장/초안 |
| **편집기** | 인플레이스 편집, 미리보기, 자동저장 |
| **온보딩** | 빈 상태에서 첫 행동 유도, 점진적 노출 |
| **설정** | 그룹핑, 검색, 위험 작업 분리 |
| **빈/오류 상태** | empty/loading/error 설계, 복구 액션 |
| **반복 업무 흐름** | 자주 하는 작업의 최단 경로, 단축키, 일괄처리 |

**기록 규칙**: UX 후보는 "이 흐름이 우리 핵심 반복 업무를 어떻게 빠르게 하는가"로 평가한다.
예쁨이 아니라 **작업 효율**이 기준.

## 3. 제품 유형별 대표 UX 패턴

`references.json` 후보를 고를 때, 제품 유형에 맞는 UX 패턴 후보가 최소 1개는 포함돼야 한다.
(ReferenceProvider `productType` 값과 정합: crm | content | commerce | admin | mobile ...)

| 제품 유형 | 대표 비교 화면(compare-ui) | 핵심 UX 패턴 | 정보 밀도 경향 |
|-----------|---------------------------|--------------|----------------|
| **CRM / 업무 SaaS** | 대시보드 · 목록 · 상세 · 폼 | left-sidebar, 마스터-디테일, 테이블+필터, 인라인 편집, 일괄작업 | medium-high~high |
| **콘텐츠 / 미디어** | 홈 · 목록 · 상세 · 검색 | 읽기 흐름, 분류·태그, 전역 검색, 추천, 가독성 우선 | low~medium |
| **커머스** | 상품목록 · 상품상세 · 장바구니 · 주문 | 필터·정렬 그리드, 상품 갤러리, 장바구니/체크아웃, 재고 상태 | medium |
| **관리자 / 어드민** | 대시보드 · 목록 · 상세 · 폼 | 고밀도 테이블, 다중 필터, 일괄작업, 권한·감사 로그, 빈/오류 상태 | high |
| **모바일 앱** | 홈 · 기능 · 상세 · 설정 | bottom-tab, 단열 reflow, 터치 타깃 ≥44px, 제스처, 핵심 기능 우선 | low~medium |
| **분석/데이터** | 대시보드 · 차트 · 테이블 · 필터 | 드릴다운, 시계열, 비교 뷰, 데이터 내보내기 | high |
| **소셜/커뮤니티** | 피드 · 상세 · 작성 · 프로필 | 피드 스크롤, 작성 흐름, 알림, 상호작용 | low~medium |

> 비교 화면은 `compare-ui` 가 동일 콘텐츠로 렌더한다(CONTRACT §13). 제품 유형 → 비교 화면 세트 매핑은 위 표를 따른다.

## 4. 후보 평가 9기준 (CONTRACT §11)

각 후보는 9개 기준으로 평가한다. 그중 5개는 `references.json` 의 `scores`(0~100)로 정량화된다.

| # | 기준 | 무엇을 묻나 | scores 필드 |
|---|------|------------|-------------|
| 1 | **제품 적합성** | 이 스타일/패턴이 제품 목적을 지원하나 | `productFit` |
| 2 | **사용자 적합성** | 우리 사용자(숙련도·맥락)에 맞나 | (브랜드/UX 종합 → 보고서) |
| 3 | **핵심 업무 적합성** | 핵심 반복 업무를 빠르게 하나 | `uxFit` |
| 4 | **정보 밀도 적합성** | 제품의 밀도 요구와 맞나 | `uxFit` 반영 |
| 5 | **브랜드 인상 적합성** | 원하는 인상(신뢰/따뜻/전문 등)과 맞나 | `brandFit` |
| 6 | **구현 가능성** | CSS Var/Tailwind로 현실적으로 구현 가능한가 | `implementation` |
| 7 | **반응형 적용 가능성** | 모바일 reflow가 가능한 구조인가 | `implementation` 반영 |
| 8 | **접근성** | 색 비의존·대비·포커스·터치를 해치지 않나 | (보고서 명시) |
| 9 | **기존 후보와의 차별성** | 다른 후보와 시각적으로 충분히 다른가 | `distinctiveness` |

`scores` 5종(productFit, brandFit, uxFit, distinctiveness, implementation)은 필수.
나머지 4기준(사용자/밀도/반응형/접근성)은 `REFERENCE-REPORT.md` 본문에 서술로 남긴다.

### 점수 해석 가이드

- 90~100: 강하게 부합, 핵심 채택 후보
- 75~89: 부합, 일부 요소 채택
- 60~74: 조건부 — 특정 요소만 또는 변형 후 채택
- <60: 부적합 — 후보에서 제외하거나 "왜 안 맞는지" 대조용으로만

## 5. 후보 다양성 규칙 — 서로 시각적으로 구별

후보는 **3~7개**(기본 5, 사용자가 직접 제안한 경우 3~5). 핵심은 **후보끼리 시각적으로 충분히
달라야 한다**는 것. 비슷한 후보 5개는 "선택"이 아니라 "착시"다.

### 권장 다양성 세트 (CONTRACT §4.2 예시)

서로 다른 방향을 대표하도록 구성:

| 후보 슬롯 | 대표 방향 | 예 |
|-----------|-----------|-----|
| A | Warm Editorial | 따뜻한 뉴트럴 + serif display + 읽기 흐름 |
| B | Compact Productivity | 고밀도 테이블 + sans + left-sidebar |
| C | Minimal Monochrome | 무채색 절제 + 강한 위계 + 무장식 |
| D | Friendly Consumer | 둥근 모서리 + 따뜻 accent + subtle-elevation |
| E | Technical Industrial | sans+mono + flat-border + 기능 semantic 색 |

### 다양성 강제 체크 (후보 확정 전)

- [ ] 두 후보의 **색 방향**이 같지 않다(둘 다 "뉴트럴+1 accent"여도 hue·톤이 구별).
- [ ] 두 후보의 **밀도**가 같지 않거나, 같으면 타이포/레이아웃이 명확히 다르다.
- [ ] 두 후보의 **타이포 family 방향**이 겹치면 위계·표면으로 차별화된다.
- [ ] 후보 중 최소 1개는 명확한 **visual-style** 대표, 최소 1개는 **ux-pattern** 대표.
- [ ] "이 후보들을 나란히 놓으면 한눈에 다른가?" → 아니면 가장 약한 후보를 다른 방향으로 교체.

`scores.distinctiveness` 는 후보 간 차별성을 반영한다. 두 후보가 너무 닮았으면 둘 다 점수가 깎인다.

## 6. 한 서비스 통째 복제 금지 (CONTRACT §18)

레퍼런스는 **원칙을 빌리는 것**이지 외형을 복제하는 것이 아니다.

### 금지

- 특정 서비스의 화면을 픽셀 단위로 따라 만들기.
- 로고·고유 일러스트·마스코트·브랜드 카피·고유 폰트 복사.
- 한 서비스의 색+타이포+레이아웃+컴포넌트를 통째로 이식.

### 의무

- 후보 선정 시 **최소 2개 근거(서비스)** 를 검토한다(`apply`/`doNotApply` 로 분리 기록).
- 레퍼런스에서 가져온 **원칙**과 우리 **실제 구현**을 분리해 `design/REFERENCES.md` 에 매핑한다
  (예: "출처: X의 '색=상태 신호 원칙' → 적용: 우리 semantic 토큰 체계").
- 최종 디자인에 **프로젝트 고유 요소**(고유 색조·고유 레이아웃 변주·고유 데이터 시각화)를 포함한다.
- 시각 스타일과 UX 구조를 **서로 다른 레퍼런스**에서 가져와 단일 복제를 구조적으로 차단한다.

### references.json 작성 시 복제 방지 체크

- [ ] `source.name/url` 명시(출처 추적 가능).
- [ ] `apply[]` = 원칙·요소 수준이지 "이 화면 그대로"가 아니다.
- [ ] `doNotApply[]` = 가져오지 않을 것 명시(통째 이식 차단).
- [ ] 최종 합성(`selection.json`)이 단일 후보 100%면 → `/design-select` 에서 고유 변주를 더하도록 안내.

## 7. 탐색 우선순위 (Provider, CONTRACT §12)

레퍼런스 소스는 다음 우선순위로 사용한다(`providers/`):

```
공식 MCP(stitch 등) > 브라우저 자동화 > 사용자가 준 URL > 사용자가 준 이미지 > 수동 등록
```

- 가용한 상위 소스를 먼저 쓰되, 미가용이면 graceful 하게 다음 단계로.
- 환상 API/존재하지 않는 서비스를 지어내지 않는다. 실제 탐색한 출처만 `source` 에 기록.
- 캡처는 `design/research/screenshots/` 에 저장하고 `candidate.screenshots[]` 에 상대경로로 연결.

## 8. `references.json` 후보 작성 체크리스트

후보 하나를 commit 하기 전:

- [ ] `id` = candidate-a..g, 후보 3~7개
- [ ] `name` = 방향을 드러내는 이름(예: "Warm Editorial")
- [ ] `source{name,url}` 실재 출처
- [ ] `category` = visual-style / ux-pattern / hybrid (§0 두 축)
- [ ] `keywords[] / recommendedFor[]`
- [ ] `strengths[] / risks[]`
- [ ] `apply[] / doNotApply[]` (원칙 수준, 복제 아님 — §6)
- [ ] `expectedImpression` 1~2단어 인상
- [ ] `screenshots[]` = screenshots/ 상대경로
- [ ] `scores{productFit, brandFit, uxFit, distinctiveness, implementation}` 0~100
- [ ] 다양성 체크(§5) 통과 — 다른 후보와 시각적으로 구별
- [ ] visual-style·ux-pattern 후보가 각각 최소 1개 존재(두 축 분리, §0)

> **요지**: 레퍼런스는 사용자가 *취향을 발견*하는 비교 재료다. 베끼는 카탈로그가 아니라,
> "이건 좋고 저건 싫다"를 말하게 하는 잘 구별되는 선택지를 만드는 것이 이 단계의 목적이다.
