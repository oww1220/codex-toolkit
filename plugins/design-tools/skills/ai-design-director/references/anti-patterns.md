# AI 디자인 안티패턴 카탈로그 (20종)

> 이 문서는 `/design-system` 이 프로젝트별 `design/ANTI-PATTERNS.md` 를 만들 때, 그리고
> `/design-audit` 가 화면을 채점할 때(감사 category `anti-pattern`) 참조하는 지식 베이스다.
> BUILD CONTRACT §7 의 20종을 각각 **증상 / 왜 생기는가 / 제품에 끼치는 해 / 대신 무엇을 / 예외 허용 조건** 으로 펼친다.

---

## 적용 원칙 — "금지 목록"이 아니라 "질문"

이 카탈로그는 특정 시각 요소를 **절대 금지하지 않는다**. 그라데이션도, 카드도, 글로우도, 큰 히어로도
적재적소에서는 옳다. 안티패턴이 되는 순간은 단 하나 — **그 요소가 제품 목적이나 사용자 행동을
어떻게 돕는지 답할 수 없을 때**다.

매 요소 앞에서 던지는 단일 질문:

> **"이 요소가 제품 목적 / 사용자 행동을 어떻게 돕는가?"**

답할 수 있으면 쓴다. 답이 "예뻐서" / "허전해서" / "요즘 다들 그렇게 해서" / "AI 가 자동으로 그렇게
줘서" 면 안티패턴이다. 즉 이것은 **사후 검열(validator)이 아니라 사전 게이트(질문)** 다.

### 왜 이 패턴들이 자동으로 튀어나오는가 (공통 근본 원인)

20종 대부분은 개별 실수가 아니라 **하나의 메커니즘**에서 파생된다 — AI(그리고 빠르게 작업하는 사람)는
명시적 방향이 없으면 **학습 분포의 최빈값**으로 회귀한다. "현대적 SaaS 랜딩 페이지" 의 평균 이미지가
곧 보라 그라데이션 + 카드 그리드 + 거대 히어로 + kicker 라벨이다. 방향(페르소나·제품 목적·선택된
레퍼런스)을 확정하지 못한 빈자리를 이 평균값이 채운다. **그래서 해법은 항상 "요소 금지"가 아니라
"방향 확정"이다.**

### 5개 예외 허용 조건 (CONTRACT §7)

아래 중 **하나라도** 명확히 충족하면, 해당 패턴은 더 이상 안티패턴이 아니다. 단 충족 근거를
`design/DECISION-LOG.md` 또는 `design/REFERENCES.md` 에 1줄로 남긴다.

| # | 예외 조건 | 확인 방법 |
|---|-----------|----------|
| 1 | 선택한 디자인 시스템에 **명시적으로 포함** | `design/DESIGN.md` / `TOKENS.json` 에 해당 토큰·규칙 존재 |
| 2 | **브랜드 정체성**과 직접 연결 | `DESIGN-BRIEF.md` 브랜드 섹션 + `REFERENCES.md` 매핑 |
| 3 | **정보구조 / 상태 전달**에 실제로 필요 | 그 요소를 빼면 사용자가 정보를 못 읽거나 상태를 오인 |
| 4 | **사용자 선택**으로 승인 | `selection.json` likes[] 또는 AskUserQuestion 승인 기록 |
| 5 | **접근성·사용성에 무해**(오히려 도움) | 대비·포커스·터치 타깃·reduced-motion 점검 통과 |

### 치명도 등급

감사(`audit/issues.json`)에서 안티패턴은 severity 를 갖는다. 아래 표의 "치명 가능"으로 표시된
항목이 근거 없이 등장하면 `critical`/`high` 로 잡고, 통과 조건(CONTRACT §9: "AI 안티패턴 치명
항목 0")을 막는다.

## 1. 의미 없는 보라/파랑 그라데이션

| 항목 | 내용 |
|------|------|
| **증상** | 버튼·헤더·배경에 목적 없이 깔린 violet→indigo / blue→cyan 사선 그라데이션. Tailwind `from-purple-500 to-indigo-600` 가 그대로. |
| **왜 생기는가** | "현대적 SaaS" 의 분포 최빈값. 색이 의미(상태·위계·브랜드)와 무관하게 *분위기*로만 쓰임. AI 기본 팔레트가 보라/파랑 계열로 편향. |
| **제품에 끼치는 해** | 색상이 의미 신호 능력을 잃는다(어디가 중요한지 색으로 구분 불가). 브랜드 무개성. 대비/가독성 저하. 다크모드에서 깨짐. |
| **대신 무엇을** | accent 1색 원칙(`color.accent.default`). 색은 **상태·위계**를 표현할 때만. 배경은 accent hue 로 살짝 tint 한 단색(`color.background.base`). 강조가 필요하면 그라데이션 대신 weight·size·여백. |
| **예외 허용** | 브랜드 그라데이션이 정체성(조건 2)이거나, 데이터 시각화에서 값의 연속을 표현(조건 3). 단 `TOKENS.json` 에 명시(조건 1). |

> 치명 가능. 근거 없는 풀스크린 보라 그라데이션 = Gate 4(Anti-Generic) 즉시 실패.

## 2. 모든 콘텐츠를 카드로

| 항목 | 내용 |
|------|------|
| **증상** | 문단·통계·링크·목록 항목까지 전부 `border + radius + shadow` 박스에 들어감. 화면이 카드의 바다. |
| **왜 생기는가** | 카드는 "정리된 느낌"을 싸게 준다. 무엇을 묶을지 판단하는 대신 모든 것을 동일 그릇에 담는 회피. |
| **제품에 끼치는 해** | 위계 소실(모두 같은 무게). 테두리·그림자가 시각 잡음. 밀도 낮아져 스캔 비효율. 테이블·목록으로 충분한 데이터를 카드화하면 비교가 어려움. |
| **대신 무엇을** | 카드는 **독립적이고 클릭 가능한 단위**(상품, 대화, 프로젝트)에만. 나열·비교 데이터는 **테이블/목록 + 구분선·여백**으로 묶음 표현. 그룹핑은 카드 대신 헤딩·여백·배경 톤 차이로. |
| **예외 허용** | 본질적으로 카드형 콘텐츠(갤러리, 칸반, 상품 그리드 — 조건 3). 선택 레퍼런스가 카드 UX(조건 1·4). |

## 3. 반복되는 3열 카드 그리드

| 항목 | 내용 |
|------|------|
| **증상** | "기능 1 / 기능 2 / 기능 3" 같은 크기·같은 구조 3개(또는 4개) 카드가 아이콘+제목+한 줄로 균등 나열. |
| **왜 생기는가** | 랜딩 "Features" 섹션의 클리셰. 정보 위계를 만들지 않고 빈칸을 메우는 가장 쉬운 레이아웃. |
| **제품에 끼치는 해** | 눈이 어디를 봐야 할지 모름(균등 = 위계 0). filler 콘텐츠("feature one/two/three")를 유발. 모든 섹션이 똑같아 보임(패턴 14·18 동반). |
| **대신 무엇을** | **비대칭 위계** — dominant 1개 + 작은 secondaries. 중요도 차이를 size·위치·여백으로. 정말 동급 3요소면 카드가 아니라 인라인 리스트나 비교 테이블. |
| **예외 허용** | 진짜로 동등한 선택지(요금제 3종 비교 — 조건 3). 단 이때도 추천 플랜은 강조(비대칭). |

## 4. 과도하게 둥근 모서리

| 항목 | 내용 |
|------|------|
| **증상** | 모든 요소 `border-radius: 16~24px+`. 버튼은 알약(pill), 입력창·카드·이미지까지 큰 곡률. |
| **왜 생기는가** | "친근/말랑" 트렌드의 무비판 적용. radius 토큰을 단계화하지 않고 큰 값 하나를 전역 적용. |
| **제품에 끼치는 해** | 업무용·전문 도구에서 장난감 같은 인상(신뢰 저하). 조밀한 데이터 UI에서 공간 낭비. 모서리만 둥글고 위계는 그대로 → 가독성 개선 없음. |
| **대신 무엇을** | `radius` 토큰 단계화(none/sm/md/lg/full). 제품 톤에 맞춤 — 업무 도구는 sm~md, 입력/컨테이너 일관. pill 은 태그·필터칩 등 의미 있는 곳만. |
| **예외 허용** | 소비자·캐주얼 브랜드 톤(조건 2·4). `design-profile.json.radius` 가 large 로 확정(조건 1). |

## 5. 근거 없는 글래스모피즘

| 항목 | 내용 |
|------|------|
| **증상** | 반투명 + `backdrop-blur` 패널이 콘텐츠 위에 떠 있음. 배경이 비쳐 텍스트 대비 불안정. |
| **왜 생기는가** | "고급스러워 보이는" 트렌드. 실제 레이어 깊이(겹침)가 없는데도 장식으로 blur 를 씀. |
| **제품에 끼치는 해** | 텍스트 가독성·대비 붕괴(접근성 실패). GPU 비용·저사양 기기 성능 저하. 다크/라이트·다양한 배경에서 일관성 깨짐. |
| **대신 무엇을** | 불투명 surface + 명확한 border/elevation 토큰. 깊이가 필요하면 `shadow` 토큰으로 정직하게. blur 는 실제 모달 오버레이 backdrop 처럼 **레이어가 진짜 겹칠 때**만. |
| **예외 허용** | 실제 떠 있는 오버레이(모달 backdrop, 미디어 위 컨트롤 — 조건 3)이고 대비 통과(조건 5). |

## 6. 장식용 글로우

| 항목 | 내용 |
|------|------|
| **증상** | 요소 뒤 색 번짐(neon glow), 버튼 box-shadow 색 발광, 배경 광원 blob. 의미 없는 빛. |
| **왜 생기는가** | "AI/미래적" 비주얼의 디폴트. 포커스·hover 같은 상태 신호와 무관하게 항상 켜져 있음. |
| **제품에 끼치는 해** | 시각 잡음으로 콘텐츠를 압도(패턴 13). 실제 상태 표시(focus glow)와 혼동. 대비·집중 저하. |
| **대신 무엇을** | 발광은 **상태 전달**에만(focus ring, 활성 토글). 강조는 색·weight·여백으로. 배경은 차분하게. |
| **예외 허용** | 게임·엔터·뮤직 등 톤이 발광을 요구(조건 2·4)하고 포커스 신호와 구분됨(조건 5). |

## 7. 화면 대부분을 차지하는 히어로

| 항목 | 내용 |
|------|------|
| **증상** | 첫 화면 80~100vh 를 거대 헤드라인+서브카피+버튼이 점유. 스크롤해야 실제 콘텐츠 시작. |
| **왜 생기는가** | 마케팅 랜딩 관습을 **업무용 앱/대시보드에 그대로** 가져옴. 첫인상=큰 글씨라는 오해. |
| **제품에 끼치는 해** | 업무 도구에서 핵심 작업이 fold 아래로 밀림(효율 저하). 정보 밀도 낭비. 마케팅/업무 패턴 혼동(Gate 1 실패). |
| **대신 무엇을** | 제품 화면은 **첫 화면에 실제 작업**(목록·요약·검색)을 둔다. 헤더는 간결. 히어로는 마케팅 페이지 한정. |
| **예외 허용** | 실제 마케팅/랜딩 페이지(조건 3)이거나, 온보딩 첫 화면처럼 단일 핵심 행동 유도(조건 3). |

## 8. 모든 제목 위 작은 영문 라벨 (kicker)

| 항목 | 내용 |
|------|------|
| **증상** | 모든 섹션·헤딩 위에 작은 대문자 영문 eyebrow("OUR FEATURES", "WHY US", "CH.03"). |
| **왜 생기는가** | 에디토리얼/랜딩 클리셰. 위계를 만드는 대신 장식 라벨을 습관적으로 붙임. |
| **제품에 끼치는 해** | 노이즈 + 번역/현지화 비용. 한국어 UI에 영문 kicker = 부조화. 모든 섹션 동일 구조화(패턴 18). 헤딩 영역과 충돌. |
| **대신 무엇을** | 위계는 heading 크기·weight 로. 분류가 필요하면 의미 있는 카테고리 텍스트(현지 언어). kicker 는 정말 분류 신호가 필요한 1~2곳만. |
| **예외 허용** | 에디토리얼 톤 레퍼런스에서 분류 신호로 채택(조건 1·4)되고 현지화됨. |

## 9. 제품과 무관한 추상 구체/입자

| 항목 | 내용 |
|------|------|
| **증상** | 배경에 3D 구(blob), 떠다니는 입자, 추상 메시 그라데이션. 제품 내용과 무관. |
| **왜 생기는가** | "비어 보이는" 공간 공포 + 트렌디한 히어로 비주얼 디폴트. |
| **제품에 끼치는 해** | 의미 0의 장식이 로딩·집중·접근성을 갉아먹음. 콘텐츠보다 먼저 보임(패턴 13). 모바일 성능 저하. |
| **대신 무엇을** | 의도적 여백(패턴 12와 균형). 시각이 필요하면 **실제 제품 데이터**(차트·실 스크린샷·실사 이미지). 추상이 필요하면 의미 있는 단 1개 flourish. |
| **예외 허용** | 브랜드 비주얼 모티프로 정의(조건 1·2)되고 콘텐츠를 안 가림(조건 5). |

## 10. 모든 버튼에 아이콘

| 항목 | 내용 |
|------|------|
| **증상** | "저장 💾", "취소 ✕", "다음 →" 처럼 모든 버튼/모든 헤딩 옆에 아이콘. |
| **왜 생기는가** | "버튼은 아이콘이 있어야 완성" 이라는 습관. 아이콘 선택이 텍스트보다 쉬워 보임. |
| **제품에 끼치는 해** | 의미 없는 아이콘이 스캔을 방해(아이콘≠즉시 이해). 잘못된 메타포는 오해 유발. 시각 잡음·일관성 붕괴. |
| **대신 무엇을** | 아이콘은 **즉시 인지 + 반복 사용 + 공간 절약**이 필요한 곳만(검색·필터·삭제 등 표준 액션). primary 액션은 텍스트만으로 충분하면 텍스트만. 아이콘은 의미 명확한 표준 세트 통일. |
| **예외 허용** | 아이콘 없이 모호한 액션(조건 3), 또는 공간 극히 제약된 툴바(조건 3). |

## 11. 과도한 그림자

| 항목 | 내용 |
|------|------|
| **증상** | 모든 요소가 떠 있음. 카드·버튼·입력창마다 크고 짙은 drop-shadow. 다중 레이어 그림자 남발. |
| **왜 생기는가** | "입체감=고급" 오해. elevation 을 단계화하지 않고 큰 그림자 하나를 전역 적용. |
| **제품에 끼치는 해** | 시각 잡음·깊이 위계 혼란(모두 떠 있으면 무엇이 위인지 모름). 조밀 UI에서 답답함. 다크모드에서 그림자 무력화. |
| **대신 무엇을** | `shadow` 토큰 단계화(none/sm/md/lg)를 **실제 레이어**(드롭다운·모달·팝오버)에만. 평면 구획은 border 로. flat-with-border 서피스 선호. |
| **예외 허용** | 떠 있는 레이어(오버레이·드래그 중 요소 — 조건 3). 디자인 시스템이 elevation 중심(조건 1). |

## 12. 과도한 여백

| 항목 | 내용 |
|------|------|
| **증상** | 한 화면에 몇 줄, 거대한 빈 공간. 스크롤해야 다음 정보. "미니멀" 이라며 정보를 숨김. |
| **왜 생기는가** | "미니멀=여백" 의 단순 등치. 밀도 판단(조밀 vs 여유)을 페르소나에서 결정하지 않은 결과. |
| **제품에 끼치는 해** | 업무용·데이터 도구에서 스캔·비교 비효율(스크롤 폭증). 정보 밀도 부족(Gate 1). 패턴 9(공간 공포)와 정반대로 동일하게 실패. |
| **대신 무엇을** | 여백은 **의도적**이어야 한다 — 그룹 구분·읽기 리듬을 위해. 밀도는 `design-profile.json.information_density` 로 확정. 업무 도구는 medium-high~high 지향. 빈 공간은 "왜 비었는가" 답할 수 있어야. |
| **예외 허용** | 집중형 콘텐츠(읽기·작성·단일 입력 — 조건 3). 여백이 정보 위계를 만드는 경우(조건 3). |

## 13. 콘텐츠보다 장식이 먼저 보임

| 항목 | 내용 |
|------|------|
| **증상** | 첫 시선이 그라데이션·일러스트·애니메이션에 먼저 꽂히고, 실제 정보·액션은 나중에 발견됨. |
| **왜 생기는가** | 장식 요소(1·6·9)가 누적되어 시각 무게가 콘텐츠를 초과. 위계 설계 부재. |
| **제품에 끼치는 해** | 사용자가 목적(정보·작업)에 도달하는 시간 증가. 전환·효율 저하. 6대 원칙 위반("목적이 장식보다 우선"). |
| **대신 무엇을** | **눈 추적 우선순위 = 정보 우선순위**. dominant 시각 무게를 핵심 콘텐츠/주요 액션에 배정. 장식은 보조(은은히 인식 못 할 수준). "자막을 잠깐 가려도 화면만으로 읽히는가" 식의 자가 테스트. |
| **예외 허용** | 거의 없음. 장식이 곧 콘텐츠인 경우(아트·포트폴리오 작품 자체 — 조건 3)만. |

> 치명 가능. Gate 4 직결.

## 14. 모바일 = 데스크톱 단순 축소

| 항목 | 내용 |
|------|------|
| **증상** | 모바일에서 데스크톱 레이아웃이 그대로 작아짐. 사이드바·다열 테이블·고정폭이 화면 밖으로. 터치 영역 과소. |
| **왜 생기는가** | 반응형을 "비율 축소"로 오해. 모바일 정보구조를 따로 설계하지 않음. |
| **제품에 끼치는 해** | 가로 스크롤·잘림·탭 불가(Gate 6 실패). 핵심 기능 모바일 사용 불가. 터치 타깃 <44px(접근성 실패). |
| **대신 무엇을** | 모바일은 **재배치**(reflow) — 사이드바→하단탭/햄버거, 테이블→카드/스택, 다열→단열. 터치 타깃 ≥44px. 핵심 기능 유지. breakpoint 토큰 기반 설계. |
| **예외 허용** | 없음(이건 거의 항상 결함). 데스크톱 전용 도구라 명시(조건 1)된 경우만 모바일 비대상. |

> 치명 가능. Gate 6 직결.

## 15. 실제 상태 없는 정적 예시 화면

| 항목 | 내용 |
|------|------|
| **증상** | "행복 경로"만 그린 목업. 빈 상태·로딩·오류·권한없음·긴 텍스트·0건/대량 데이터가 없음. |
| **왜 생기는가** | 한 장의 예쁜 스크린샷에 최적화. 상태 설계는 노동이라 회피. |
| **제품에 끼치는 해** | 실제 사용에서 무너짐(빈 화면·깨진 레이아웃·복구 불가 오류). 사용자 신뢰 상실. `/design-prototype` 의 상태 요구 위반. |
| **대신 무엇을** | 핵심 화면마다 **상태 세트**: empty(첫 사용/검색 0건) · loading(스켈레톤) · error(원인+복구 액션) · 권한/제약 · 데이터 극단(긴 문자열, 대량 행). 빈 상태는 다음 행동 안내. |
| **예외 허용** | 없음. 단 발췌 예시(`examples/`)는 행복 경로 일부만 보여도 됨(데모 목적 명시). |

## 16. 의미 없는 통계 숫자

| 항목 | 내용 |
|------|------|
| **증상** | "99% 만족", "10x 빠르게", "100% 보장", "50K+ 사용자" 같은 출처·근거 없는 수치. |
| **왜 생기는가** | "신뢰=숫자" 의 값싼 대체. 진짜 데이터 대신 그럴듯한 자리표시. |
| **제품에 끼치는 해** | 거짓·과장으로 신뢰 훼손. 진짜 데이터와 구분 불가. 법적/윤리 리스크. AI slop 의 대표 신호. |
| **대신 무엇을** | **실제 데이터만**. 없으면 정직한 placeholder("아직 데이터 없음" / 회색 박스 + 라벨). "honest placeholder beats a fake stat." 수치엔 출처. |
| **예외 허용** | 검증된 실제 지표(조건 3)이고 출처 표기. 데모 데이터는 명시적 라벨. |

> 치명 가능(신뢰·법적 리스크).

## 17. 사용자 흐름 없이 카드만 배치한 대시보드

| 항목 | 내용 |
|------|------|
| **증상** | 위젯/통계 카드만 잔뜩. "그래서 다음에 뭘 하지?" 가 없음. 액션·드릴다운·우선순위 부재. |
| **왜 생기는가** | 대시보드=숫자 카드 모음이라는 관습. 사용자 작업 흐름(JTBD)에서 출발하지 않음. |
| **제품에 끼치는 해** | 보기만 하고 행동 못 하는 화면(가치 전달 실패). 패턴 2·3·17 동반. 제품 목적 미지원(Gate 1·5). |
| **대신 무엇을** | 대시보드는 **"무엇을 봐야 하고 → 무엇을 할 수 있는가"** 흐름. 핵심 지표 1~2개 dominant + 그에 연결된 액션·드릴다운. 우선순위 비대칭. |
| **예외 허용** | 순수 모니터링 화면(읽기 전용 상황판 — 조건 3). 그래도 이상 시 알림·링크는 둠. |

## 18. 모든 섹션이 동일 구조인 랜딩

| 항목 | 내용 |
|------|------|
| **증상** | 헤딩+서브+3카드(또는 좌텍스트/우이미지)가 섹션마다 똑같이 반복. 스크롤해도 같은 리듬. |
| **왜 생기는가** | 한 블록 템플릿을 복제해 페이지를 채움(template loop). 섹션별 콘텐츠 어감 차이 무시. |
| **제품에 끼치는 해** | 단조로워 읽기를 멈춤. 위계·강조 소실. 패턴 3·8 동반. "5섹션째에 패턴 인식 → 6섹션째부터 지루." |
| **대신 무엇을** | 섹션마다 **다른 레이아웃 리듬**(전폭 인용 / 비대칭 / 단일 거대 수치 / 비교 테이블 / 타임라인). 콘텐츠 성격에 맞는 구조 재발명. |
| **예외 허용** | 의도된 반복 리듬(FAQ 아코디언, 일관 리스트 — 조건 3). 단 강조 섹션은 변주. |

## 19. 너무 많은 강조색

| 항목 | 내용 |
|------|------|
| **증상** | 빨강·파랑·초록·노랑이 동시에 강조로 쓰임. 모든 것이 컬러풀해 무엇이 중요한지 불명. |
| **왜 생기는가** | accent 토큰을 1색으로 제한하지 않음. 색을 위계가 아닌 다양성으로 사용. |
| **제품에 끼치는 해** | 강조의 의미 소실(다 강조면 강조 0). 색맹·저시력 사용자 혼란. 브랜드 인상 산만. |
| **대신 무엇을** | **accent 1색 원칙**(`color.accent.default`). 추가 색은 semantic(success/warning/danger/info)으로 **의미 고정**. 강조는 색 1개 + weight/size. |
| **예외 허용** | 데이터 시각화의 카테고리 구분색(조건 3, 색맹 대응 팔레트). 색=의미가 일관된 경우. |

## 20. 일관되지 않은 모서리·간격

| 항목 | 내용 |
|------|------|
| **증상** | 버튼 radius 8px, 카드 12px, 입력창 6px. 간격이 13·17·23px 처럼 제각각. 토큰 밖 매직넘버. |
| **왜 생기는가** | `TOKENS.json` 단일 출처를 안 쓰고 컴포넌트마다 즉석 값. AI가 화면마다 다른 숫자 생성. |
| **제품에 끼치는 해** | 미묘하게 어수선(무의식적 불신). 유지보수 지옥. Gate 3(System Consistency) 실패. |
| **대신 무엇을** | 모든 radius·spacing 은 `TOKENS.json` 스케일에서만(spacing 4px 기반, radius none/sm/md/lg/full). 토큰 밖 값이 필요하면 `DECISION-LOG.md` 에 이유. `validate-tokens.ts` 로 강제. |
| **예외 허용** | 광학 보정용 미세 조정(조건 5)이고 토큰으로 승격되거나 로그에 기록. |

## 자가 점검 (화면 작성 / 감사 시작 전 30초)

> 1. 지금 이 화면에서 자동으로 손이 간 요소는 위 20종 중 무엇인가?
> 2. 각각에 대해 "제품 목적/사용자 행동을 어떻게 돕는가" 1문장으로 답할 수 있는가?
> 3. 답 못 하는 요소 → 제거하거나 5개 예외 중 하나로 정당화하고 로그에 남긴다.
> 4. 치명 가능(1·13·14·16) 항목이 근거 없이 있으면 → 감사 통과 불가, 즉시 수정.
