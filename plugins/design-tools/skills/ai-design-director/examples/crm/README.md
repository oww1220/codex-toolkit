# 예시 · 개인사업자 CRM (플래그십)

> 이 폴더는 **실행 결과 샘플이 아니다.** AI Design Director 스킬에 들어가는 *입력 시나리오*와,
> 9단계 파이프라인이 만들어내는 *핵심 산출물의 발췌*를 모아 "이 스킬이 무엇을 만드는가"를 보여준다.
> (BUILD CONTRACT §15) 분량은 발췌 수준으로 간결하되, 실제 산출물과 동일한 스키마·경로·용어를 쓴다.

---

## 시나리오 (입력)

개인사업자(1인~소규모) 한 명이 **고객·매출·경비**를 한곳에서 관리하는 웹 앱.
의뢰인은 디자인을 말로 설명하지 못하지만, 싫은 것은 분명하다.

> "회계 프로그램처럼 딱딱하고 답답한 건 싫어요. 그렇다고 요즘 스타트업 앱처럼 알록달록하고
> 둥글둥글 귀여운 것도 우리 일에는 안 맞고요. 매일 오래 들여다보는 화면이니까 **눈이 안 피곤**했으면 좋겠고,
> 숫자랑 목록이 많은데도 **복잡해 보이지 않았으면** 해요."

이 한 문단이 스킬 전체를 관통하는 설계 과제다:

| 사용자가 피하고 싶은 양극단 | 디자인이 찾아야 할 중간 |
|---|---|
| 회계 SW의 차갑고 빽빽한 표 (엑셀/세무 프로그램) | 정보는 조밀하되 **시각적으로 차분한** 밀도 |
| 스타트업/소비자 앱의 알록달록·둥글·이모지 | 따뜻하되 **절제된** 뉴트럴, 장식 최소 |
| "정보가 많다" = "복잡하다"는 통념 | 그룹핑·위계로 **많은 정보를 단순하게** 보이게 |

핵심 긴장은 **"오래 봐도 안 피곤(차분)" ↔ "정보 밀도 높음(업무 효율)"** 의 양립이다.
이 둘은 보통 충돌하지만(차분=여백 vs 밀도=빽빽), 스킬은 한쪽을 버리지 않고
*밀도는 유지하되 색·장식·모서리를 절제해 피로를 줄이는* 방향으로 조정한다.

## 9단계가 이 시나리오를 어떻게 통과하는가

스킬은 바로 코드를 만들지 않는다. **이해 → 번역 → 탐색 → 비교 → 선택 → 시스템 → 목업 → 검증**의
순서로, 매 단계에서 사용자가 *판단*하게 한다. (6대 원칙 中 "생성보다 판단")

| 단계 | 커맨드 | 이 시나리오에서 한 일 | 산출물 |
|------|--------|----------------------|--------|
| 1 | `/design-brief` | "딱딱함 싫음 / 귀여움 싫음 / 눈 안 피곤 / 복잡해 보이지 않게"를 원문 그대로 채집, 확정·추정 분리 | `DESIGN-BRIEF.md` |
| 2 | `/design-translate` | 말 → 디자인 언어. "오래 써도 안 피곤"=warm-neutral·저채도·장식 최소, "정보 많음"=medium-high density. 충돌(차분↔밀도) 조정 | `DESIGN-TRANSLATION.md` · `design-profile.json` |
| 3 | `/design-research` | 4개 후보를 **시각 스타일 축 / UX 패턴 축** 분리해 탐색 (한 서비스 통째 복제 금지) | `research/REFERENCE-REPORT.md` · `references.json` · `screenshots/` |
| 4 | `/design-compare` | 같은 화면·같은 데이터·같은 문장으로 후보를 나란히. **디자인만** 다르게 | `compare/` (React/Vite 앱) |
| 5 | `/design-select` | 사용자가 요소 단위로 고름 → 조합 합성 | `DESIGN-DIRECTION.md` · `selection.json` · `DECISION-LOG.md` |
| 6 | `/design-system` | 선택을 토큰·원칙·컴포넌트로 고정 | `DESIGN.md` · `TOKENS.json` 외 8종 |
| 7 | `/design-prototype` | 핵심 흐름 목업 + 상태(빈/로딩/오류) + 반응형 | 프런트엔드 목업 + 스크린샷 |
| 8 | `/design-audit` | 브라우저 검증 + 7영역 점수 + 개선 루프(최대 3회) | `audit/` |

> 전체를 한 번에 돌리려면 `/design-full`. 단 **선택 단계(4·5)는 자동 skip 금지** — 사용자 승인 없이 확정하지 않는다.

## 이 발견의 핵심 — 비교가 취향을 드러낸다

번역 단계에서 추정한 프로필은 *가설*일 뿐이다. 의뢰인은 후보 4개를 나란히 보고서야
자기 취향을 발견했다(원칙 2 "말보다 선택을 신뢰"):

- **Warm Editorial(A)** 의 humanist 타이포 + serif 디스플레이 짝 → "이 글씨가 안 피곤하다" (채택)
- **Compact Productivity(B)** 의 촘촘한 목록·테이블 + 좌측 사이드바 → "정보가 많은데 안 답답하다" (채택)
- **Calm Professional(C)** 의 저채도 warm-neutral 팔레트 → "딱 이 정도 따뜻함" (채택)
- **Minimal Monochrome(D)** 는 전체로는 너무 차가웠지만, **얇은 라인 아이콘 세트만** 가져옴 (부분 채택)

즉 최종 방향은 *한 후보의 복제*가 아니라 **A 타이포 + B 정보구조 + C 색상 + D 아이콘**의 조합이다
(원칙 5 "단일 또는 조합 선택"). 시각 스타일(A·C·D)과 UX 구조(B)를 **서로 다른 레퍼런스**에서
가져온 것이 이 스킬의 핵심 동작이다(원칙 3 "시각 ↔ UX 분리").

| `DESIGN-BRIEF.md` | 인터뷰 발췌 — 원문 인용 + 확정/추정 분리 | `/design-brief` |

> 순수 `#000`/`#fff` 대신 잉크/종이 톤 off-black·off-white, Tailwind 기본 보라/파랑 대신
> 절제된 따뜻한 단색 accent, 둥근 카드 그리드 대신 border 구분 테이블. 이 스킬은 *그 회귀를 막는 것이 상품*이다.
