## 1. 문체 (Writing Style)

| 규칙 | 정의 |
|------|------|
| 인칭 | 사용자 = {{"회원님" / "당신" / 호칭 생략}}, 시스템 = 1인칭 자제(주어 생략 기본) |
| 시제·태 | 능동·현재형 기본. 피동("저장되었습니다")보다 능동·완료("저장했어요" / "저장됨") |
| 문장 길이 | 한 문장 한 메시지. {{40자 내외}} 권장, 긴 설명은 문장 분리 |
| 톤 일관성 | 한 화면 안에서 존댓말·반말, 격식·구어를 섞지 않는다 |
| 명령 vs 안내 | 버튼·동작은 명령형 동사, 설명은 평서형 |

> 가이드: AI 기본값인 과장·홍보체("혁신적인", "강력한", "손쉽게")를 피한다. 사용자는 일하러 왔지
> 광고를 보러 오지 않았다. 사실·행동 중심으로 쓴다.

## 2. 존댓말 여부 (Politeness Level) — **프로젝트 전체 1회 결정, 이후 고정**

선택: **{{ 해요체("저장했어요") / 합쇼체("저장하였습니다") / 음슴체·명사형("저장됨") }}**

| 맥락 | 표기 |
|------|------|
| 안내·도움말 | {{해요체}} |
| 버튼·라벨 | {{명사형/동사원형}} ("저장", "삭제") |
| 시스템 상태 | {{명사형}} ("저장됨", "동기화 중") |
| 오류 | {{해요체}} ("불러오지 못했어요") |

**규칙**: 한 제품에서 위 셋을 **혼용하지 않는다**. 결정 근거: {{brand_impression / 사용자층}}.

> 가이드: 존댓말 단계는 브랜드 인상과 직결된다. "전문성/신뢰"가 1순위면 합쇼체·명사형이,
> "친근함"이면 해요체가 보통 맞다. 한 번 정하면 전 화면 일관 적용 — 섞이면 싸구려로 보인다.

## 3. 제목 길이 (Headings)

| 위치 | 길이 가이드 | 예 |
|------|------------|----|
| 화면 제목(H1) | {{2~6 단어 / ~20자}} | "고객 목록" |
| 섹션 제목(H2) | {{~24자}} | "이번 달 매출" |
| 카드/항목 제목 | {{~30자, 넘으면 말줄임}} | |
| 모달 제목 | {{동작 명시}} | "거래 삭제" |

**규칙**: 제목은 내용을 설명하지 장식하지 않는다. 모든 제목 위에 작은 영문 라벨(kicker)을 붙이지 않는다(안티패턴 #8). 거대 헤드라인 카피 금지(업무 도구, 안티패턴 #7).

> 가이드: 제목은 스캔용이다. 길면 못 읽고, 멋부리면 안 읽힌다. 영문 kicker 남발은 전형적 AI slop.

## 4. 버튼 문구 (Buttons & Actions)

| 규칙 | 정의 | 예(좋음) | 예(나쁨) |
|------|------|---------|---------|
| 동사+목적어 | 무엇을 하는지 명시 | "고객 추가", "변경사항 저장" | "확인", "제출", "클릭" |
| 길이 | 1~3 단어 | "거래 추가" | "새로운 거래를 추가하기" |
| 1차 동작 | 화면당 1개, 명확한 위계(`design/COMPONENTS.md`) | "저장" | (1차 버튼 2개 금지) |
| 위험 동작 | danger 색 + 명시 동사 | "영구 삭제" | "확인" |
| 취소·보조 | 중립 표현 | "취소", "나중에" | |
| 로딩 중 | 동작 진행형 | "저장 중…" | (라벨 사라짐) |

**확인 다이얼로그 버튼**: "확인/취소" 대신 **실제 행동**을 적는다("삭제"/"취소", "내보내기"/"닫기"). `design/UX-PATTERNS.md` 10. 확인 다이얼로그와 일치.

> 가이드: "확인"은 무엇을 확인하는지 알려주지 않는다. 버튼만 읽어도 무슨 일이 일어날지 알아야 한다.

## 5. 안내 · 도움 문구 (Helper & Hint Text)

| 위치 | 규칙 | 예 |
|------|------|----|
| 입력 힌트 | 형식·예시 제시(placeholder 와 구별) | "예: 010-1234-5678" |
| 필드 설명 | 왜 필요한지, 짧게 | "세금계산서 발행에 쓰여요" |
| 툴팁 | 1문장, 보조 정보만(필수 정보를 숨기지 않음) | |
| 빈 칸 안내 | placeholder 는 *예시*지 *라벨 대체가 아님*(Gate 7) | |

> 가이드: placeholder 에 라벨을 넣으면 입력 시작 순간 사라져 접근성·기억에 해롭다. 라벨은 항상 따로 둔다.

## 6. 오류 메시지 (Error Messages)

구조: **무엇이 잘못됐는지 · (가능하면) 왜 · 어떻게 고치는지.** 사용자를 탓하지 않는다.

| 상황 | 좋은 예 | 나쁜 예 |
|------|---------|---------|
| 필수 누락 | "이름을 입력해 주세요." | "입력값이 유효하지 않습니다." |
| 형식 오류 | "이메일 형식이 맞지 않아요. 예: name@example.com" | "Invalid input" |
| 저장 실패 | "저장하지 못했어요. 잠시 후 다시 시도해 주세요." | "Error 500" |
| 권한 거부 | "이 작업은 관리자만 할 수 있어요." | "Forbidden" |
| 네트워크 | "연결이 불안정해요. 자동으로 다시 시도할게요." | "Network error" |

**규칙**
- 기술 용어·에러코드·스택을 사용자에게 노출하지 않는다(필요 시 작은 참조 ID만).
- 색만으로 오류 표시 금지 — 아이콘·텍스트 병행(Gate 7, `design/UX-PATTERNS.md` 12.).
- 부정문보다 해결을 제시("~할 수 없습니다" → "~하려면 ~하세요").

> 가이드: 오류 메시지의 목적은 비난이 아니라 복구다. 읽고 나서 "그래서 뭘 하지?"가 남으면 실패.

## 7. 빈 화면 문구 (Empty States)

`design/UX-PATTERNS.md` 11. 빈 상태와 1:1 대응. 종류별로 다른 문구.

| 종류 | 문구 | CTA |
|------|------|-----|
| 최초(데이터 0) | "{{아직 등록한 고객이 없어요. 첫 고객을 추가해 보세요.}}" | "{{고객 추가}}" |
| 검색 0건 | "'{검색어}'에 대한 결과가 없어요." | (검색어 수정 안내) |
| 필터 0건 | "조건에 맞는 항목이 없어요." | "필터 초기화" |
| 권한 없음 | "이 화면을 볼 권한이 없어요." | {{문의/요청}} |

**규칙**: 빈 화면은 막다른 길이 아니라 시작점 — 항상 다음 행동 1개. 가짜 통계/가짜 예시로 채우지 않는다(안티패턴 #15,#16).

> 가이드: "데이터가 없습니다."로 끝내지 않는다. *왜 비었고 무엇을 하면 되는지*를 한 문장으로.

## 8. 성공 · 완료 메시지 (Success Feedback)

| 상황 | 문구 | 형태 |
|------|------|------|
| 저장 | "저장했어요" | 토스트(2~3초) |
| 생성 | "{{고객}}을 추가했어요" | 토스트 + (선택)다음 행동 |
| 삭제 | "삭제했어요 · 되돌리기" | 토스트(되돌리기 버튼 포함) |
| 일괄 작업 | "{{N건}} 처리했어요" | 토스트 |
| 장기 작업 완료 | "{{내보내기}}가 끝났어요" | 토스트/배너 + 결과 링크 |

**규칙**: 성공 메시지는 짧고 사실적으로. 과한 축하("축하합니다! 🎉")·이모지 장식 금지(안티패턴 — `design/TOKENS.json` 정신, 진부한 기본값 회피). 되돌릴 수 있는 동작이면 되돌리기를 함께.

> 가이드: 성공 알림은 *확인*이지 *축하*가 아니다. 매번 폭죽을 터뜨리면 업무가 산만해진다.

## 9. 날짜 · 시간 · 숫자 표기 (Dates, Numbers, Units)

| 항목 | 규칙 | 예 |
|------|------|----|
| 날짜 | {{YYYY.MM.DD / YYYY년 M월 D일}} — 전 화면 통일 | 2026.06.20 |
| 상대 시간 | {{"방금", "3분 전", "어제"}} (최근), 그 이전은 절대 날짜 | "3분 전" / "2026.06.18" |
| 시간 | {{24시간제 HH:MM / 오전·오후}} | 14:30 |
| 통화 | {{₩1,250,000 / 1,250,000원}} — 천 단위 구분, 통일 | |
| 큰 수 | {{천 단위 콤마, 필요 시 만/억 단위}} | 12,000 / 1.2만 |
| 소수·퍼센트 | {{자릿수 통일}} | 12.5% |
| 범위·단위 | 숫자와 단위 사이 띄어쓰기 규칙 통일 | "3건", "12 GB" |
| 빈 값 | {{"—" 또는 "없음"}} — 0과 구별 | |

**규칙**: 같은 종류의 데이터는 화면이 달라도 동일 형식. 가짜 수치(99%, 10x 등) 절대 사용 금지(안티패턴 #16) — 실제 데이터만 표기.

> 가이드: 표기 흔들림(어떤 화면은 "2026-06-20", 다른 화면은 "6월 20일")은 신뢰를 깎는다. 한 번 정하고 고정.

## 10. 전문 용어 · 도메인 어휘 (Terminology)

한 개념에는 한 단어. 동의어 혼용 금지(용어집을 단일 출처로).

| 개념 | 채택 용어 | 금지(혼용 금지) |
|------|----------|----------------|
| {{고객}} | {{"고객"}} | {{"클라이언트", "거래처", "유저"}} |
| {{거래}} | {{"거래"}} | {{"트랜잭션", "건"}} |
| {{삭제}} | {{"삭제"}} | {{"제거", "지우기"}} |
| {{설정}} | {{"설정"}} | {{"환경설정", "옵션", "세팅"}} |

**규칙**
- 영어 약어·전문 용어는 대상 사용자가 아는 수준에서만. 모르면 풀어쓰거나 1회 정의.
- 내비게이션·버튼·제목·도움말에서 같은 개념은 **반드시 같은 단어**(`design/UX-PATTERNS.md` 와 교차 일치).

> 가이드: 같은 것을 화면마다 다르게 부르면 사용자는 다른 것으로 오해한다. 용어집은 본 표가 단일 출처.

## 11. 금지 표현 (Banned Phrases)

| 분류 | 금지 | 대체 |
|------|------|------|
| 마케팅 과장 | "혁신적인", "강력한", "최고의", "손쉽게", "단 한 번의 클릭으로" | 사실·동작 기술 |
| 모호함 | "어떤 문제가 발생했습니다", "잠시만요" | 구체적 원인·행동 |
| 비난 | "잘못 입력하셨습니다", "올바르지 않은…" | "다시 확인해 주세요"(중립) |
| 기술 누출 | "null", "undefined", "Error 4xx/5xx", 스택트레이스 | 사람 언어로 번역 |
| 이모지 장식 | 🎉✨🚀💡🔥 등 | (사용 안 함) |
| 가짜 수치·구호 | "99% 만족", "10배 빠르게" | 실제 값 또는 생략 |
| 영문 kicker 남발 | 모든 제목 위 작은 영문 라벨 | (제거) |

> 가이드: 이 표는 이 스킬이 막으려는 "AI slop 문체"의 텍스트판이다. 위 표현이 화면에 보이면
> 자동으로 진부함으로 회귀한 신호 — `references/anti-patterns.md` 의 시각 안티패턴과 짝을 이룬다.
