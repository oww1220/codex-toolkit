# Completion Summary

## 무엇을 / 언제 (한 줄)
구현이나 수정이 끝난 뒤 완료 답변을 쉬운 설명, 판단 이유, 실제 변경, 검증 결과 중심으로 정리할 때 사용한다.

## 제약 (MUST DO / MUST NOT DO) — 스텝 위에 둔다
- ✅ 실제로 수행한 작업과 변경만 쓴다.
- ✅ 왜 그렇게 작업했는지 기존 구조, 재사용, 최소 변경, 리스크 감소 관점에서 쉽게 설명한다.
- ✅ 실행한 검증과 실행하지 못한 검증을 구분해서 쓴다.
- ⛔ 하지 않은 작업이나 확인하지 않은 효과를 완료한 것처럼 쓰지 않는다.
- ⛔ 기술 용어만 나열해서 사용자가 판단하기 어렵게 만들지 않는다.
- ⛔ 커밋 메시지 작성 요청을 대신 처리하지 않는다.
- ⛔ 사용자가 요청하지 않은 긴 보고서, 파일 산출물, changelog 생성을 하지 않는다.

## 핵심 워크플로우
1. 작업 내용을 확인한다 — 입력: 이번 턴에서 실제로 수정한 파일, 실행한 명령, 실패/생략한 검증 -> 출력: 보고할 사실 목록
2. 변경 이유를 고른다 — 입력: 기존 코드 구조, 재사용한 패턴, 선택한 최소 변경 -> 출력: 사용자가 이해할 수 있는 이유 1-3개
3. 완료 답변을 짧은 섹션과 2-5개 bullet로 작성한다 — 입력: 사실 목록과 변경 이유 -> 출력: 작업한 것, 변경 전/후, 왜 이렇게 했는지, 쉽게 풀어쓴 설명, 검증
4. 검증: 답변에 실제 변경, 변경 전/후, 작업 이유, 쉬운 설명, 검증 결과, 못 한 검증이 포함되어 있고 과장된 내용이 없는지 를 확인한다.
5. 출력: 사용자가 바로 판단할 수 있게 채팅 답변에만 정리한다.

## 출력 위치 (필수)
- ✅ `작업한 것`, `변경 전/후`, `왜 이렇게 했는지`, `쉽게 풀어쓴 설명`, `검증`을 포함한 완료 요약
- ✅ 실행하지 못한 검증이 있으면 `못 한 검증`으로 명시
- ✅ 기본은 2-5개 bullet이며, 사용자가 더 자세한 설명을 원할 때만 늘린다
- ✅ 변경 이유는 “기존 구조를 따랐다”, “중복을 피했다”, “검증 가능한 작은 변경으로 좁혔다”처럼 판단 근거로 쓴다.
- ⛔ 실행하지 않은 명령, 확인하지 않은 화면, 만들지 않은 파일을 검증이나 변경으로 쓰지 않는다.
