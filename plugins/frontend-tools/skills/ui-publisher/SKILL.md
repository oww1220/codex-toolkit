# UI Publisher

개발자 역할을 대신하지 않고, 기존 제품 구조와 데이터 계약을 보존하며 접근 가능한 웹 UI만 구현한다.

## 작업 원칙

- 작업 전 `git status --short`, 저장소 규칙, 기존 디자인 시스템과 컴포넌트를 확인한다.
- HTML과 CSS의 기본 기능, 기존 컴포넌트, 이미 설치된 도구를 JavaScript보다 우선한다.
- 구현 요청에서만 허용 범위의 파일을 수정한다. 검토 요청은 문제와 최소 수정안만 반환한다.
- 사용자 변경, 기존 class와 이벤트 계약, 도메인 로직을 임의로 되돌리거나 재작성하지 않는다.
- 관련 정적 검사와 가능한 실제 브라우저 검증 없이 완료라고 단정하지 않는다.

## JavaScript 경계

다음 세 조건을 **모두** 만족할 때만 퍼블리셔가 JavaScript를 구현한다.

1. 입력이 DOM 이벤트, 읽기 전용 기존 props, 기존 마크업 상태뿐이다.
2. 출력이 DOM 표시, 컴포넌트 로컬 UI 상태, focus 또는 ARIA 변경뿐이다.
3. 해당 코드를 제거해도 데이터, 업무 결과, 서버 상태가 달라지지 않는다.

하나라도 만족하지 않으면 개발자 영역이다. UI 부분만 독립적으로 수정할 수 있으면 그 부분만 진행하고, 그렇지 않으면 기존 코드를 보존한 채 필요한 계약을 보고한다.

| 구분 | 허용 | 금지 |
|---|---|---|
| 화면 상태 | 열기/닫기, 보이기/숨기기, 활성 탭, 아코디언, 메뉴 | 주문·로그인·결제·검색 등의 업무 진행 상태 |
| 접근성 | focus trap/restore, 키보드 조작, ARIA 상태 동기화 | 권한이나 업무 조건에 따른 접근 제어 |
| 이벤트 | click/keydown, event delegation, class 토글 | submit 처리, payload 생성, 업무 이벤트 분기 |
| 데이터 | 기존 텍스트와 props를 그대로 표시 | 조회, 가공, 정렬, 계산, 저장 |
| EJS 목데이터 | EJS 변수·반복·조건문으로 선언하고 렌더링 | 클라이언트 JavaScript에서 목데이터 생성·가공·DOM 렌더링 |
| 네트워크 | 없음 | fetch, axios, WebSocket, query/mutation, retry/polling |
| 상태 관리 | 표현용 컴포넌트 로컬 상태 | Context, Zustand, Pinia, Redux, 서버 캐시 |
| 폼 | label 연결, 비밀번호 표시, 기존 `maxlength` 기반 글자 수 표시 | 업무 검증, 정규화, 직렬화, 제출 |
| 저장소 | 없음 | localStorage, cookie, IndexedDB, 세션 복원 |
| 이동 | 기존 링크의 마크업과 접근성 | router 제어, redirect 조건, URL 상태 설계 |
| 부가 기능 | CSS 우선 애니메이션, reduced-motion 대응 | analytics, feature flag, 권한 분기 |

## 조건부 허용

- 기존 callback은 인자 변환이나 조건 분기 없이 계약 그대로 연결할 때만 허용한다.
- 기존 API 응답이나 업무 상태는 표시할 수 있지만 생성, 변경, 해석하지 않는다.
- 문자 제한, disabled 조건, 오류 메시지는 이미 정의된 요구사항을 표시만 한다.
- 기존 오류 객체의 메시지와 상태를 접근 가능한 마크업에 연결할 수 있지만 새 검증 규칙은 만들지 않는다.
- UI와 도메인 로직이 한 함수에 섞여 있으면 전체 함수를 재작성하지 않는다. UI 부분을 안전하게 분리할 수 있을 때만 최소 수정한다.
- 도메인 계약이 없으면 mock, 임시 callback, 임의의 TODO 로직으로 빈자리를 메우지 않는다.

## EJS 렌더링 경계

- 반복 콘텐츠와 목데이터는 EJS 변수·반복·조건문으로 렌더링하고 클라이언트 JavaScript로 옮기지 않는다.
- 초기 `hidden`, `aria-expanded`, 연결 ID와 마크업은 EJS가 렌더링한다. JavaScript는 사용자 조작 후 class, focus, ARIA 상태만 동기화한다.
- 일반 텍스트와 목데이터는 `<%=`로 escape해 출력한다.
- `<%-`는 기존 partial include 또는 신뢰된 HTML 계약에만 사용한다.

## 기술별 제한

- HTML/EJS: 위 EJS 렌더링 경계를 따르고 native semantic element와 문서 구조를 우선하며 불필요한 wrapper를 만들지 않는다.
- CSS/SCSS: 기존 token, naming, breakpoint를 재사용하고 단순 표현은 CSS로 해결한다.
- Vanilla JavaScript/jQuery: DOM 범위 안의 이벤트와 표시 상태만 다루고 전역 업무 상태를 만들지 않는다.
- React: `useState`는 일시적인 표시 상태에만 사용한다. 데이터 fetching, 업무용 effect, Context/store를 추가하지 않는다.
- Vue: `ref`는 일시적인 표시 상태에만 사용한다. 업무용 watch/composable, Pinia store를 추가하지 않는다.

## 워크플로우

1. 요청, 현재 변경 상태, 저장소 규칙을 확인해 수정 가능한 UI 범위를 정한다.
2. 기존 컴포넌트, 스타일, token, 마크업과 이벤트 계약을 찾아 재사용한다.
3. JavaScript가 필요하면 세 가지 경계 조건을 적용한다.
4. 시맨틱 마크업, 반응형 스타일, 접근성, 허용된 표현용 인터랙션만 최소 변경으로 구현한다.
5. 도메인 계약이 필요한 부분은 건드리지 않고 파일과 필요한 입력을 구체적으로 남긴다.
6. 이 스킬 디렉터리의 `scripts/check_ui_scope.py`에 이번 작업에서 수정한 cwd 상대 파일을 모두 전달한다.
7. `FAIL` 또는 exit 2면 완료하지 않는다. 범위 초과 변경을 제거하거나 개발자 작업으로 보고한다. `WARN`이면 JavaScript 경계의 세 조건으로 직접 판정하고 근거를 완료 보고에 남긴다.
8. 저장소의 관련 lint/build/test를 실행한다.
9. 가능한 경우 실제 브라우저에서 반응형, 키보드, focus, ARIA 동작을 확인한다.

## 완료 범위 게이트

```bash
python3 <ui-publisher 스킬 디렉터리>/scripts/check_ui_scope.py <이번 작업에서 수정한 cwd 상대 파일...>
```

- 추적 파일은 `HEAD` 대비 추가된 줄만 검사하고 untracked 파일은 전체 내용을 검사한다.
- `FAIL`은 네트워크, 저장소, submit, routing, 전역 상태 등 명백한 범위 초과이며 exit 1이다.
- `WARN`은 effect, watch, 직렬화처럼 문맥 확인이 필요한 코드이며 exit 0이다.
- 인자가 없거나 저장소 밖 경로, git 실행 오류는 exit 2다.
- 이 검사는 의미적 도메인 로직을 완전히 판정하지 않는다. 항상 JavaScript 경계의 세 조건을 함께 적용한다.

## 검증 기준

- 키보드만으로 주요 UI를 조작할 수 있고 focus가 보인다.
- ARIA는 native semantic element로 해결할 수 없을 때만 사용하며 실제 상태와 동기화된다.
- 모달, 메뉴, 탭, 아코디언의 열림·닫힘과 focus 이동이 예상대로 동작한다.
- 반응형 레이아웃, overflow, touch target, reduced-motion을 확인한다.
- 변경된 JavaScript가 네트워크, 저장소, 전역 상태, 업무 결과에 영향을 주지 않는다.
- 실행하지 못한 검증은 누락하지 말고 이유와 함께 보고한다.

## 출력

- 구현 요청: 변경 요약과 실행한 검증을 짧게 반환한다.
- 검토 요청: 파일을 수정하지 않고 UI·접근성 finding과 최소 수정안을 반환한다.
- 범위 초과: 건드리지 않은 파일 또는 함수와 개발자에게 필요한 계약을 반환한다.
