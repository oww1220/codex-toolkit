## 1. 정보 구조 (Information Architecture)

제품의 객체(엔티티)와 그 관계, 그리고 그것이 화면 계층으로 어떻게 나뉘는가.

| 항목 | 정의 | 근거 |
|------|------|------|
| 1차 객체 | {{예: 고객, 거래, 청구서}} | {{브리프의 핵심 작업}} |
| 2차 객체 | {{예: 메모, 첨부, 활동 로그}} | |
| 최상위 화면(섹션) | {{예: 대시보드 / 고객 / 매출 / 경비 / 설정}} | {{사용자가 가장 자주 오가는 작업 단위}} |
| 화면 깊이 | 최대 {{N}}단계 (예: 섹션 → 목록 → 상세 → 편집) | 깊이가 4단계를 넘으면 길 잃음 위험 |
| 진입점(랜딩) | {{로그인 후 첫 화면}} | {{가장 빈번한 작업 또는 요약}} |

**규칙**
- 한 화면은 한 가지 주 목적을 갖는다. 목록 화면에 상세 편집 폼을 욱여넣지 않는다(안티패턴 #17).
- 객체 간 이동은 양방향이어야 한다(고객 → 거래, 거래 → 고객). 막다른 화면 금지.
- 같은 객체의 목록·상세·편집은 동일한 라벨·아이콘·용어를 쓴다(`design/CONTENT-STYLE.md` 와 일치).

> 가이드: 정보구조는 "예쁜가"가 아니라 "사용자가 머릿속 모델대로 찾는가"로 검증한다.
> 의심되면 카드소팅 결과나 사용자 발언(브리프)을 근거로 인용한다.

## 3. 핵심 사용자 흐름 (Critical Flows)

`design/DESIGN-DIRECTION.md` 의 핵심 흐름을 단계별로 옮긴다. **흐름당 1개 표.**

### 흐름 A — {{예: 새 고객 등록 후 첫 거래 입력까지}}

| # | 사용자 행동 | 화면/상태 | 시스템 반응 | 다음 행동 유도 |
|---|------------|-----------|------------|---------------|
| 1 | {{고객 추가 버튼}} | {{고객 목록}} | {{생성 폼 열림(모달/페이지)}} | {{필수 입력에 포커스}} |
| 2 | {{이름·연락처 입력 후 저장}} | {{생성 폼}} | {{저장 → 토스트 → 상세로 이동}} | {{"거래 추가" 1차 동작 제시}} |
| 3 | {{거래 추가}} | {{고객 상세}} | {{거래 폼}} | … |

**흐름 설계 규칙**
- 각 단계 끝에서 사용자는 "다음에 뭘 하지?"를 고민하지 않아야 한다(다음 1차 동작을 화면이 제시).
- 반복 업무는 단계 수를 최소화한다. 불필요한 확인 클릭을 넣지 않는다(Gate 5).
- 흐름 중단·이탈 지점(브라우저 닫음, 뒤로가기)에서 입력 손실을 막는다(8. 저장 참조).

> 가이드: 흐름은 "한 번에 끝나는가"로 검증한다. 3단계로 끝날 일을 6단계로 만들면 안티패턴.

| 구성 | 제목(무엇을) · 본문(결과·영향) · 1차/2차 버튼 |
| 버튼 문구 | 동사 명시("삭제", "내보내기") — "확인/취소" 대신 행동을 적음(`design/CONTENT-STYLE.md`) |
| 위험 동작 | 1차 버튼을 danger 색 + 오른쪽, 기본 포커스는 안전한 쪽 |
| 닫기 | Esc·배경 클릭으로 취소 가능(파괴 동작은 배경 클릭 무시 고려) |

> 가이드: 확인창은 "이걸 정말 할 것인가"가 아니라 "무슨 일이 일어나는가"를 알려주는 도구다.

## 11. 빈 상태 (Empty States)

데이터가 **아직 없을 때**의 화면. 정적 가짜 예시(안티패턴 #15)·의미 없는 일러스트로 채우지 않는다.

| 빈 상태 종류 | 메시지 | 1차 행동 |
|-------------|--------|---------|
| 최초(데이터 0) | "{{아직 등록한 고객이 없어요}}" | "{{첫 고객 추가}}" 버튼 |
| 검색 0건 | "{{'{검색어}' 결과가 없습니다}}" | 검색어 수정 안내 |
| 필터 0건 | "{{조건에 맞는 항목이 없습니다}}" | "필터 초기화"(5. 참조) |
| 권한 없음 | "{{이 화면을 볼 권한이 없습니다}}" | {{요청/문의}} |
| 오류로 비었음 | 12. 오류 복구로 처리(빈 상태 아님) | 재시도 |

**규칙**: 빈 상태는 *막다른 길이 아니라 시작점*. 항상 다음 행동 1개를 제시한다.
일러스트는 의미 있을 때만(브랜드·정보 전달). 장식용 추상 구체·입자 금지(안티패턴 #9).

> 가이드: 빈 화면을 가짜 데이터로 채워 "있어 보이게" 만들지 않는다(안티패턴 #15,#16). 정직한 빈 상태가 낫다.

## 12. 오류 처리 · 복구 (Error & Recovery)

| 오류 종류 | 표현 위치 | 복구 |
|----------|----------|------|
| 입력 검증 | 해당 필드 인라인 + 첫 오류로 포커스 | 수정 후 즉시 재검증 |
| 저장 실패 | 토스트/배너 + 입력값 보존 | "다시 시도" |
| 네트워크 | 비차단 배너("연결이 불안정합니다") | 자동 재시도 + 수동 재시도 |
| 권한 거부 | 인라인 메시지 | {{요청 경로}} |
| 404/없는 객체 | 전용 화면 | "{{목록으로}}" |
| 치명적(앱 깨짐) | 폴백 화면 + 오류 ID | 재로드 |

**규칙**
- 오류 메시지는 **무엇이·왜·어떻게**를 담는다(원문 스택/코드 노출 금지). 문구는 `design/CONTENT-STYLE.md`.
- 색만으로 오류 표시 금지 — 아이콘·텍스트 병행(Gate 7).
- 사용자 입력은 오류가 나도 사라지지 않는다.

> 가이드: 좋은 오류는 사용자를 탓하지 않고 다음 행동을 알려준다. "오류가 발생했습니다"만 띄우면 안티패턴.

## 13. 권한 · 역할 (Permissions & Roles)

`design/DESIGN-BRIEF.md` 의 권한 모델을 반영. 단일 사용자 제품이면 "권한 모델 없음"으로 명시.

| 역할 | 볼 수 있음 | 할 수 있음 | UI 처리 |
|------|-----------|-----------|---------|
| {{관리자}} | {{전체}} | {{전체}} | — |
| {{멤버}} | {{본인+공유}} | {{생성·편집}} | 비허용 동작은 {{숨김 / 비활성+사유}} |
| {{뷰어}} | {{읽기}} | {{없음}} | 편집 진입점 숨김 |

**규칙**
- 권한 없는 동작은 {{기본적으로 숨김}}. 비활성으로 둘 경우 이유를 툴팁으로 제공.
- 권한 없는 화면 직접 접근 시 11. 빈 상태(권한 없음)로 처리.
- 클라이언트 숨김은 UX일 뿐 보안이 아님 — 서버 검증을 전제(구현 주의 표기).

> 가이드: "숨김 vs 비활성"을 일관되게 정한다. 어떤 곳은 숨기고 어떤 곳은 비활성이면 혼란.

## 14. 모바일 동작 (Mobile Behavior)

데스크톱을 그냥 축소하지 않는다(안티패턴 #14). **재배치**한다.

| 영역 | 데스크톱 | 모바일(<640px) |
|------|---------|----------------|
| 내비게이션 | {{좌측 사이드바}} | {{하단 탭 / 햄버거}} |
| 테이블 | 다열 | {{핵심 컬럼만 카드형 행 / 가로 스크롤(고정 첫 컬럼)}} |
| 필터 | 인라인 바 | {{바텀시트}} |
| 1차 동작 | 상단 버튼 | {{하단 고정 버튼 / FAB}} |
| 상세 패널 | 분할 보기 | 전체 화면 + 뒤로가기 |
| 폼 | 다열 | 1열, 적절한 input type·키보드 |

**규칙**
- 터치 타깃 ≥ 44px(Gate 6,7). 가로 스크롤로 콘텐츠 잘림 금지.
- 핵심 기능은 모바일에서도 유지(숨겨서 못 쓰게 하지 않음).
- `design/TOKENS.json` 의 `breakpoint` 값을 기준으로 분기.

> 가이드: 모바일에서 "이 기능은 PC에서 하세요"는 최후의 수단. 핵심 흐름(3.)은 모바일에서 끝나야 한다.

## 15. 키보드 사용 (Keyboard)

| 동작 | 키 | 적용 범위 |
|------|----|----|
| 포커스 이동 | Tab / Shift+Tab — 시각적 순서와 일치 | 전역 |
| 활성화 | Enter / Space | 버튼·링크 |
| 닫기/취소 | Esc | 모달·드롭다운·검색 |
| 목록 이동 | ↑ ↓ | 목록·검색결과·메뉴 |
| 빠른 실행 | {{Cmd/Ctrl+K}} | 커맨드 팔레트(쓰는 경우) |
| 저장 | {{Cmd/Ctrl+S}} | 편집 화면(쓰는 경우) |

**규칙**
- 모든 인터랙티브 요소는 키보드로 도달·조작 가능(Gate 7).
- 포커스 표시(focus ring)는 항상 보여야 한다 — `outline:none` 단독 금지.
- 모달 열림 시 포커스 가둠(focus trap), 닫을 때 직전 위치 복귀.
- 단축키는 발견 가능해야 한다(툴팁·도움말에 표기).

> 가이드: 키보드만으로 핵심 흐름(3.)을 끝까지 수행해 본다. 한 곳이라도 막히면 접근성 결함.
